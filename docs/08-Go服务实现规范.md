## 0. 统一骨架

```
backend/go/cmd/<svc-name>/
└── main.go                       # 仅装配：读配置、初始化依赖、启动

backend/go/internal/<svc-name>/
├── config.go                     # Viper 配置结构 + 验证
├── server.go                     # Connect/gRPC 服务器装配
├── service.go                    # 业务实现（RPC handlers）
├── repo/                         # 数据访问
├── adapter/                      # 外部依赖封装
└── domain/                       # 领域模型 + 状态机
```

### 0.1 公共库（`backend/go/internal/common/`）

| 包 | 内容 |
|---|---|
| `common/config` | Viper + fsnotify 热加载 |
| `common/logger` | zap，注入 trace_id/tenant_id |
| `common/telemetry` | OTel + Prom 初始化 |
| `common/db/pg` | pgx 连接池 + 事务工具 + RLS 上下文 |
| `common/db/ch` | clickhouse-go v2 封装 |

| `common/auth` | JWT 验证、tenant ctx 提取 |
| `common/interceptor` | Connect/gRPC 通用 interceptor |

| `common/vault` | Vault Agent 客户端 |
| `common/idempotency` | Redis idempotency key 工具 |

### 0.2 main.go 模板

```go
func main() {
    cfg := config.Load[Config]("<svc>")
    log := logger.New(cfg.Log)
    ctx, cancel := signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGTERM)
    defer cancel()

    shutdown := telemetry.Init(ctx, cfg.Telemetry)
    defer shutdown()

    deps := buildDeps(ctx, cfg)
    srv := server.New(cfg, deps)

    if err := srv.Run(ctx); err != nil {
        log.Fatal("server exit", "err", err)
    }
}
```

### 0.3 所有服务必须

- 启动时打印版本、commit、配置摘要（脱敏）
- 暴露 `/healthz` `/readyz` `/metrics`
- 平滑关闭：收到 SIGTERM 后停止接受新请求，处理完在途 30s 超时
- 配置变更：fsnotify 检测，可热加载的字段单独标记 `reloadable`
- 所有 RPC 装载完整 interceptor 链

## 1. trading-core

### 1.1 职责

唯一对前端暴露的 Connect 服务。承载所有 §03 中前端 RPC。

### 1.2 目录

```
backend/go/cmd/trading-core/main.go
backend/go/internal/adminapi/
├── config.go
├── server.go
├── auth/
│   ├── service.go               # AuthService 实现
│   ├── jwt.go
│   ├── totp.go
│   └── password.go
├── tenant/
├── user/
├── account/
├── broker/
├── symbol/
├── factor/
├── strategy/
├── order/
├── position/
├── risk/
├── backtest/
├── notify/
├── audit/
└── stream/                       # SSE 流（订阅 NATS 转发）
```

### 1.3 Interceptor 顺序

```
otel → logger → recover → auth → tenant_guard → permission → audit → rate_limit → idempotency → handler
```

### 1.4 关键实现点

- **登录**：argon2id 校验密码，TOTP 校验，签 JWT，写 refresh_token 到 cookie
- **JWT**：EdDSA（Ed25519），kid 轮换支持，黑名单 Redis
- **PG RLS 上下文**：每个 handler 入口 `SET LOCAL app.tenant_id = '...'`，事务结束自动清理
- **流式 RPC**：订阅 NATS subject（按 tenant/account 过滤），转 `stream.Send()`；客户端断开自动取消订阅
- **审计**：写 RPC 完成后异步写 `audit_logs`（用 NATS 解耦避免阻塞响应）

### 1.5 配置

```yaml
trading-core:
  http:
    addr: :8080
    tls: {cert_file, key_file}    # 生产从 Vault PKI 注入
  jwt:
    issuer: alfq
    access_ttl: 15m
    refresh_ttl: 168h
  cookie:
    domain: app.alfq.io
    secure: true
    same_site: strict
  pg:    {dsn_env: PG_DSN, max_conn: 50}
  ch:    {addr: clickhouse:9000, database: alfq}
  redis: {addr: redis:6379}
  nats:  {url: nats://nats:4222}
  vault: {addr: http://vault:8200}
```

## 2. md-gateway

### 2.1 职责

接 mtapi `Streams.OnQuote` / `OnTickValue` 流，标准化 → 发布到 NATS，落盘到 ClickHouse。

### 2.2 目录

```
backend/go/internal/mdgateway/
├── config.go
├── server.go
├── manager.go                    # 连接管理（每 broker 多连接池）
├── connection.go                 # 单 mtapi 连接生命周期
├── normalizer.go                 # mtapi msg → alfq.v1.Tick
├── publisher.go                  # 发 NATS + 落 CH
├── healthcheck.go                # 定期 CheckConnect
└── adapter/
    ├── mt4/                      # MT4 gRPC 客户端封装；导入 backend/go/gen/mt4 (来自 mt4grpc3.mtapi.io:443)
    └── mt5/                      # MT5 gRPC 客户端封装；导入 backend/go/gen/mt5 (来自 mt5grpc3.mtapi.io:443)
                                  # 桩代码生成见 doc 25 §3；两 package 独立，不共享 proto 类型
```

### 2.3 连接策略

- 每 `accounts.is_active=true` 行一个连接
- 启动顺序：`Connect / ConnectEx` → `Subscribe(symbols)` → `OnQuote(id)` 循环消费
- 心跳：每 15s 调 `CheckConnect`，失败指数退避重连（1s/2s/4s/...，最大 60s）
- 断线超 5 分钟告警 P1
- 优雅停机：`Disconnect` 后关闭 stream

### 2.4 数据流

```
mtapi OnQuoteReply
  → normalize 成 alfq.v1.Tick (含 tenant_id/broker/symbol/ts/bid/ask)
  → fanout:
       NATS publish "md.tick.<broker>.<symbol>"
       CH batch insert (异步, 每 1s 或 1000 条 flush)
  → Redis set "quote:<broker>:<symbol>" (最新快照)
```

### 2.5 背压

- CH 写入失败：缓冲到本地磁盘队列（badger / pebble），恢复后回放
- NATS 慢：丢弃旧 tick（行情可丢），告警

### 2.6 指标

- `md_tick_total{broker,symbol,tenant}`
- `md_tick_latency_seconds`（broker ts → 接收 ts）
- `md_connection_state{broker,account}` (gauge)
- `md_reconnect_total{broker,account}`

## 3. quant-engine

### 3.1 职责

订阅行情/bar 流，按已激活因子的 DSL 增量计算，发布因子值。

### 3.2 目录

```
backend/go/internal/factorsvc/
├── config.go
├── server.go
├── registry.go                   # 因子注册表（从 PG 加载）
├── engine.go                     # 增量计算引擎
├── dsl/                          # DSL 解析与执行（详见 09）
│   ├── parser.go
│   ├── ast.go
│   ├── compiler.go
│   └── ops/                      # ema/sma/std/atr/ref/...
└── publisher.go                  # 输出 NATS + CH
```

### 3.3 计算模型

- 每个 (tenant, factor, symbol, period) 一个**有状态算子树**实例
- 每来一根 bar（订阅 `md.bar.<broker>.<symbol>.<period>`）：
  1. 找到该 (factor, symbol) 的算子树
  2. `eval(bar)` 增量推进状态
  3. 发布 `factor.<name>.<symbol>` (tenant 路由 key)，落 CH

### 3.4 启动 / 热更新

- 启动：从 `factors` 表加载 `status=approved` 的所有因子
- 监听 PG NOTIFY（或定时拉取 + version 列）感知变更
- 变更：增量重建算子树（不影响其他因子）

### 3.5 策略子组件 (strategysvc)

#### 职责

加载策略部署、订阅因子流、执行 Spec、生成信号 → 提交风控/OMS。

#### 目录

```
backend/go/internal/strategysvc/
├── config.go
├── server.go
├── loader.go                     # 从 PG 加载 deployments
├── runner.go                     # 单部署 goroutine
├── spec/                         # 策略 spec 解析
│   ├── parser.go
│   └── validate.go
├── signal/                       # signal 模块（dsl_rule / onnx_threshold / ...）
│   ├── onnx.go                   # github.com/yalue/onnxruntime_go
│   ├── threshold.go
│   └── linear.go
├── sizing/                       # 仓位算法
├── exit/                         # 出场规则
└── client/                       # trading-core / oms gRPC client
```

#### Runner

```
每个 deployment 一个 goroutine：
  - 订阅相关 factor 主题 + bar 主题
  - 每收一组完整的 factor 值 → 推理 → 决定 target_qty
  - 若与当前仓位不一致 → 构造 OrderRequest → 调 trading-core.Check → 通过则提交 OMS
  - 监听 NATS "order.evt.*" 更新内部仓位状态
  - 监听 NATS "risk.cmd" 接收 Kill 指令立即停止
```

#### ONNX 加载

- 部署启动时从 MinIO 下载 ONNX 到本地 cache
- 校验文件 hash 与 PG 中记录一致（防篡改）
- onnxruntime-go 加载，复用 session 实例

### 1.6 风控子组件 (risksvc)

#### 职责

订单必经检查点；维护熔断状态；触发 Kill Switch。

#### 目录

```
backend/go/internal/risksvc/
├── config.go
├── server.go
├── service.go                    # RiskCheckService 实现
├── rules/                        # 规则引擎
│   ├── max_lot.go
│   ├── max_position.go
│   ├── daily_loss.go
│   ├── drawdown.go
│   ├── reject_rate.go
│   ├── slippage.go
│   ├── session.go                # 交易时段
│   ├── whitelist.go
│   └── registry.go               # 规则注册 + 加载
├── state/                        # 实时状态（账户 PnL、回撤等）
├── breaker/                      # 熔断器
└── killswitch/
```

#### 检查流程

```
RiskCheck(req):
  1. 加载所有适用规则（global / account / strategy）
  2. 顺序执行，任何 deny → 立即返回 reject
  3. 全部通过 → 返回 approved
  4. 写 risk_events 异步
```

#### 状态来源

- 订阅 `order.evt.*` 维护实时仓位 / 已实现 PnL
- 订阅 `md.tick.*` 维护浮动 PnL
- 每分钟拉 `account_balance_snapshots` 校准

#### Kill Switch

```
KillSwitch(scope, by_user, totp):
  1. 校验权限 + TOTP
  2. 发 NATS "risk.cmd" {action: HALT, scope}
  3. quant-engine 收到立即停止生成新信号
  4. 根据策略：撤所有挂单 / 平所有仓
  5. 写 risk_events critical + 告警
```

### 1.7 OMS 子组件 (oms)

#### 职责

订单状态机 + 调 BrokerAdapter + 对账 + 持仓更新。

#### 目录

```
backend/go/internal/oms/
├── config.go
├── server.go
├── service.go                    # OrderService 内部 + PlaceOrder API
├── statemachine/                 # pending/submitted/.../filled
├── repo/                         # orders/positions PG 访问
├── reconciler/                   # 对账（每分钟）
└── adapter/                      # BrokerAdapter 客户端
```

#### 状态机

> **唯一源**：订单完整状态机定义在 doc 14 §3.1，上下文不变量在 doc 14 §3.2。本节仅给 OMS 实现视角的简化型（同名状态，有向上总结）：

```
[new]
  │ validate (参数/权限/品种可交易)
  ▼
[validated] ─────fail────► [rejected]
  │ risk_check
  ▼
[risk_approved] ───fail──► [rejected]
  │ submit → broker
  ▼
[submitted] ──────broker_reject/error──► [failed]
  ├─ working                            (限价/挂单等触发)
  ├─ fill_partial ─► [partially_filled] ─├─ fill_full ─► [filled]
  ├─ fill_full   ─► [filled]                └─ cancel    ─► [cancelled]
  ├─ cancel       ─► [cancelled]
  └─ expire       ─► [expired]
```

终态集合：`{rejected, failed, cancelled, expired, filled}`。任何终态后收到 broker 更新 → 告警 + 对账（见 doc 14 §7.2）。

#### 幂等

- 入参 `client_order_id` 唯一索引于 `(account_id, client_order_id)`
- Redis NX 锁 5s 防并发重复
- 重试用同一 `client_order_id`，OMS 内部根据状态决定补发或返回历史结果

#### Broker Adapter

```
BrokerAdapter interface {
  Submit(ctx, OrderRequest) (BrokerResp, error)
  Cancel(ctx, BrokerTicket) error
  Modify(ctx, BrokerTicket, ...) error
  Query(ctx, BrokerTicket) (Order, error)
  StreamUpdates(ctx, AccountID) (<-chan OrderUpdate, error)
}
```

实现 `MT4Adapter` / `MT5Adapter`（调 `gprc/mt4.proto` / `mt5.proto`）。未来加 FIX/CTP 实现同接口。

> ⚠️ **MT4 与 MT5 是两套不可共用的协议**：proto schema、枚举语义（`Op` vs `OrderType` / `Execution` vs `FillPolicy`）、订单/Deal 模型、撮合规则均不同。**禁止**复用同一 mapper，必须为每个平台维护独立的 `internal/broker/mt4/` 与 `internal/broker/mt5/` 子包；仅共享 `BrokerAdapter` 接口与领域类型（`Order/Position/Fill`），**不共享 proto 类型**。详见 `docs/14-领域模型与交易规则.md` §3.4。

#### 对账

```
每 60s：
  pull mtapi OpenedOrders + last hour OrderHistory
  compare with PG orders by ticket
  diff → 写 risk_events warn，超阈值告警
```
