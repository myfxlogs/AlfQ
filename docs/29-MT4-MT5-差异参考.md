# 29 - MT4 / MT5 网关能力差异参考

> **目的**：避免 AI Agent 在实现 symbol 元数据 / 行情 / 订单时把 MT4 和 MT5 当同一个东西。
> **单一事实源**：本文内容直接来自 `backend/go/gen/mt4/` 与 `backend/go/gen/mt5/` 的生成代码，由 AI Agent 检索 proto stub 确认。
> **更新规则**：发现新的差异立即更新本文，**不要**把 MT4/MT5 差异写在 README 或 MASTER-ROADMAP。
>
> 关联文档：`docs/25-Proto代码生成与gRPC调用规范.md` · `docs/tasks/MASTER-ROADMAP.md` §2

---

## 1. 核心结论

> ⚠️ **MT4 与 MT5 在 gRPC 接口、字段命名、嵌套结构、单位惯例上全部不同。任何"用一个抽象层就能搞定"的实现都会带来隐藏 bug。**

正确做法：
1. **两边各自实现 fetcher**（mt4_fetcher.go / mt5_fetcher.go）
2. **共用同一个落库目标结构**（`broker_symbols` 表）—— 由 fetcher 各自负责字段映射
3. **共用 canonical 算法**
4. **共用错误处理 / 重试 / 超时策略**

错误做法：
- 写一个统一 client interface 试图屏蔽差异（字段缺失/语义不同 → 数据腐烂）
- 把 MT5 的字段名直接当成 MT4 的字段（很多字段在 MT4 里不存在或名字不同）

## 2. Symbol 元数据 RPC 对比

### 2.1 RPC 列表

| 能力 | MT5 | MT4 | 说明 |
|---|---|---|---|
| 列出所有 symbol 名 | `Symbols(SymbolsRequest)` 返回完整参数 | `Symbols(SymbolsRequest)` 仅返回 `[]string` | **返回类型不同**：MT5 给参数，MT4 给名字列表 |
| 单 symbol 详情 | `SymbolParams(symbol)` → `SymbolParamsReply` | `SymbolParams(symbol)` → `SymbolParamsReply` | 字段结构差异巨大，见 §2.2 |
| 批量 symbol 详情 | `SymbolParamsMany(limit)` | `SymbolParamsMany()` | MT5 支持 limit，MT4 无 |
| 仅 symbol 名字列表 | `SymbolList` | 用 `Symbols` 替代 | MT4 无独立 SymbolList |
| 交易时段 | `SymbolSessionsEx` / `SymbolSessionsExMany` | **无独立 RPC** | MT4 把 sessions 嵌入在 `ConGroupSec.Sessions` 里 |
| 行情 + 实时 spread | `MarketWatchMany` | **无** | MT4 用 `Quote` / `GetQuoteMany` |
| 服务器时区 | `ServerTimezone` | `ServerTimezone` | 同 |
| Tick 价值 / 大小 | 内嵌 `SymbolParams.tick_value` | `TickValueWithSize` 单独 RPC | MT4 单独取 |

### 2.2 SymbolParams 字段对比

#### MT5 — `SymbolParamsReply.Result` 直接是 `SymbolInfoEx`

字段路径：`Result.{Field}`（**扁平**）

| 字段 | 路径 | 类型 |
|---|---|---|
| 名称 | `SymbolName` | string |
| 小数位 | `Digits` | int32 |
| 最小变动 | `Point` | float64 |
| 合约规模 | `ContractSize` | float64 |
| Tick size | `TickSize` | float64 |
| Tick value | `TickValue` | float64 |
| 最小手数 | `VolumeMin` | float64 |
| 最大手数 | `VolumeMax` | float64 |
| 手数步长 | `VolumeStep` | float64 |
| 初始保证金 | `MarginInitial` | float64 |
| 多头 swap | `SwapLong` | float64 |
| 空头 swap | `SwapShort` | float64 |
| 三倍 swap 日 | `SwapRollover3Days` | int32 |
| 保证金币种 | `CurrencyMargin` | string |
| 盈亏币种 | `CurrencyProfit` | string |
| 基础币种 | `CurrencyBase` | string |
| 交易模式 | `TradeMode` | enum (0 disabled, 1 long, 2 short, 3 full) |
| 描述 | `Description` | string |

#### MT4 — `SymbolParamsReply.Result` 是 `SymbolParams`（三层嵌套）

字段路径：
- 基础参数：`Result.Symbol.{Field}` —— `SymbolInfo`
- 分组配置：`Result.Group.{Field}` —— `ConSymbolGroup`
- 分组细则：`Result.GroupParams.{Field}` —— `ConGroupSec`（**真正的合约规模 / sessions / margin 在这里**）

| 字段 | 路径 | 类型 |
|---|---|---|
| 名称 | `Result.SymbolName` | string |
| 小数位 | `Result.Symbol.Digits` | int32 |
| 最小变动 | `Result.Symbol.Point` | float64 |
| 多头 swap | `Result.Symbol.SwapLong` | float64 |
| 空头 swap | `Result.Symbol.SwapShort` | float64 |
| 合约规模 | `Result.GroupParams.ContractSize` | float64 |
| Tick value | `Result.GroupParams.TickValue` | float64 |
| Tick size | `Result.GroupParams.TickSize` | float64 |
| 初始保证金 | `Result.GroupParams.MarginInitial` | float64 |
| 维持保证金 | `Result.GroupParams.MarginMaintenance` | float64 |
| 保证金计算模式 | `Result.GroupParams.MarginMode` | int32 |
| 盈亏计算模式 | `Result.GroupParams.ProfitMode` | int32 |
| 三倍 swap 日 | `Result.GroupParams.SwapRollover3Days` | int32 |
| 交易时段 | `Result.GroupParams.Sessions[]` | `[]*ConSessions`（7 元素，按周日开始） |
| 最小/最大/步长 lot | `Result.GroupParams.{LotMin,LotMax,LotStep}` | float64 | （在更大的 ConGroupSec 中，字段名按 proto） |
| 仅多头 | `Result.GroupParams.LongOnly` | int32 (0/1) |
| Bid tick value | `Result.GroupParams.BidTickvalue` | float64 |
| Ask tick value | `Result.GroupParams.AskTickvalue` | float64 |

> **关键**：MT4 缺少独立的 `min_lot/max_lot/lot_step` 简单字段——需要查 `GroupParams` 中对应 proto 定义（部分字段名为 `lot_min` / `lot_max` / `lot_step`，部分经纪商不提供）。
> **关键**：MT4 的 `Sessions` 是按日的 `[]*ConSessions`，每天可能有多个 [open, close] 时段，需要遍历展开。

### 2.3 字段在两边都缺失但需要补的

| 需求 | MT4 / MT5 是否给 | 补救 |
|---|---|---|
| 交易日历（节假日） | 否 | 自建 `holiday_calendar` 表，按 broker_id 关联 |
| 合约到期（CFD/期货） | MT5 部分给 `Expiration`；MT4 不可靠 | 单独维护到期表 |
| Index/股票 split 调整 | 否 | 外部数据源 |

## 3. 行情 / Bar / 历史数据 RPC 对比

| 能力 | MT5 | MT4 | 备注 |
|---|---|---|---|
| 实时报价订阅 | `Streams.OnQuote` | `Streams.OnQuote` | API 相似但 reply 结构不同 |
| 实时 Order Update | `Streams.OnOrderUpdate` | `Streams.OnOrderUpdate` | 已用 |
| 历史 K 线（bar） | `CopyRates` / `CopyRatesRange` | `QuoteHistory` | **名字完全不同** |
| 历史 tick | `CopyTicks` / `CopyTicksRange` | **无原生**（部分经纪商不提供） | MT4 多数情况只能从 quote stream 自录 |
| 多 symbol 报价批量 | `GetQuoteMany` / `MarketWatchMany` | `GetQuoteMany` | 字段不同 |
| OHLCV 单价时间 | bar 内含 `tick_volume` + `real_volume` | bar 仅含 tick volume | volume 含义不同 |

### 3.1 Bar (K 线) 结构差异

**MT5 `MqlRates`**:
```
time, open, high, low, close, tick_volume, spread, real_volume
```

**MT4 `QuoteHistory.Rate`**:
```
time, open, high, low, close, volume
```

> MT4 `volume` 实质是 tick_volume，不是真实成交量。**回测时若依赖 real volume，MT4 不可用**。

## 4. 订单 / 持仓 RPC 对比

| 能力 | MT5 | MT4 | 备注 |
|---|---|---|---|
| 当前持仓 | `OpenedOrders` 含 `Position` + `Order` 两种 | `OpenedOrders` 全部是 `Order` | MT5 区分 position / pending order，MT4 不区分 |
| 历史 | `OrderHistory` 仅订单 / `PendingOrderHistory` / `DealHistory` | `OrderHistory` 合一 | MT5 有 deals (成交记录)；MT4 没有 deals 概念 |
| 下单 | `OrderSend` (含 `position_id`) | `OrderSend` | MT5 用 ticket + position_id 双标识 |
| 平仓 | `PositionClose` / `OrderSendCloseBy` | `OrderClose` | MT5 平仓走 position id |
| 修改 | `OrderModify` | `OrderModify` | 字段不同 |
| 拒单原因 | 详细 `Retcode` | 简单 `Result` 字符串 | MT4 排错难度高 |

### 4.1 Order Type 枚举差异

**MT4** `Op`：
```
0 Op_Buy
1 Op_Sell
2 Op_BuyLimit
3 Op_SellLimit
4 Op_BuyStop
5 Op_SellStop
6 Op_Balance     ← 出入金
7 Op_Credit
```

**MT5** `OrderType`：
```
0  Buy
1  Sell
2  BuyLimit
3  SellLimit
4  BuyStop
5  SellStop
6  BuyStopLimit
7  SellStopLimit
8  CloseBy
... Balance/Credit 在 OrderType_Balance/Credit 单独枚举
```

> **拉历史时**：MT4 必须过滤 type ≥ 6 排除出入金；MT5 同理但枚举值不同。

## 5. Sessions（交易时段）特别说明

### 5.1 MT5

独立 RPC：`SymbolSessionsEx(symbol)` → `SymbolSessionsExReply`
返回：
```
quotes: 7 个 SessionInfo（按周日→周六）
trades: 7 个 SessionInfo
```
每个 SessionInfo 含数个 `[open_seconds, close_seconds]` 区间。

### 5.2 MT4

嵌入式：`SymbolParams.GroupParams.Sessions[]`（`[]*ConSessions`，7 元素）
每元素含 quote_sessions 与 trade_sessions 两个数组。

**采集时差异处理**：
```go
// MT5
sessions, _ := mt5.SymbolSessionsEx(...)
storeSessions(sessions.Quotes, sessions.Trades)

// MT4
params, _ := mt4.SymbolParams(...)
storeSessions(extractMT4Quotes(params.GroupParams.Sessions),
              extractMT4Trades(params.GroupParams.Sessions))
```

## 6. 平台能力判别（运行时）

某些经纪商对 MT4 关闭了部分 RPC（例如某些 RPC 返回空）。落地时**必须**：

1. 每次拉取后校验关键字段（`digits`, `point`, `contract_size`）是否非零
2. 任一字段为 0 → 写告警日志，标记 symbol 为 `partial`，**禁止下单**
3. 提供降级路径：核心字段缺失时尝试从 `Quote` / `TickValueWithSize` 补全

## 7. AI Agent 实施 checklist

实现 SM-1 / SM-2 / SM-3 等任务时：

- [ ] 阅读本文，**不要假设**两边接口一致
- [ ] MT4 fetcher 走 `Result.Symbol` + `Result.GroupParams` 双路径取字段
- [ ] MT5 fetcher 走 `Result` 扁平路径
- [ ] **每个落库字段都明确标注来源**（fetch 时记录 `source: "mt5/SymbolParams.ContractSize"` 之类的元信息到 `raw_payload`）
- [ ] 字段缺失 / 为零时不要悄悄写入，要日志 + 计数器
- [ ] 单元测试用真实经纪商返回的 fixture（`testdata/mt4_*.json` / `testdata/mt5_*.json`）
- [ ] 对账：同一经纪商 MT4 vs MT5 账户拉到的 `EURUSD` 参数差异要落到 metrics 监控

## 8. 已知历史坑

| 坑 | 影响 | 规避 |
|---|---|---|
| MT4 `Sessions` 偶尔返回全空 | 无法判断当下能否交易 | 用 `ServerTimezone` + 通用 FX 时段表兜底 |
| MT4 `MarginInitial` 部分经纪商为 0 | 仓位计算错误 | 从 `Symbol.MarginCalcMode` + `ContractSize` 推算 |
| MT5 `Symbols(SymbolsRequest)` 返回字段不全 | 用 `SymbolParamsMany` 替代 | 不依赖 `Symbols` |
| MT4 `Op_Balance/Credit` 进历史 | 出入金被当订单 | type ≥ 6 过滤（已在 `mtapi/client.go` 实现） |
| MT5 `Position` 与 `Order` 不区分会重复计仓 | 计仓错误 | 严格区分 ticket vs position_id |
| MT4 服务器时间多为 EET/UTC+2 / +3 | bar 边界错配 | `ServerTimezone` + 本地存 UTC |

## 9. 参考实现位置

| 模块 | MT5 | MT4 |
|---|---|---|
| 连接 | `backend/go/internal/mdgateway/adapter/mt5/` | `backend/go/internal/mdgateway/adapter/mt4/` |
| 账户 / 历史 | `backend/go/internal/mdgateway/adapter/mtapi/client.go`（统一抽象） | 同 |
| 行情 stream | `backend/go/internal/mdgateway/gateway_mt5.go` | `backend/go/internal/mdgateway/gateway_mt4.go` |
| Symbol 同步 | 待新增 `backend/go/internal/symbolsync/mt5_fetcher.go` | 待新增 `backend/go/internal/symbolsync/mt4_fetcher.go` |

`mtapi/client.go` 已经对账户/订单这类**强对齐**的 API 做了 platform switch 抽象，可以参考其分支写法。但 symbol 元数据**字段差异更大**，建议各自独立实现，不要硬塞进同一函数。
