# 13 - 参考项目研习指南（先抄后改）

> **本文件让"先抄后改"落地**。AI Agent 在实施每个模块前必须：
> 1. 用脚本克隆对应开源项目到 `references/`
> 2. 按本文"阅读路线"读指定文件/范围
> 3. 在自己写的模块 README 顶部用 `Reference:` 字段标明来源
> 4. 实施完成后在 PR 描述里说明"借鉴了什么、改了什么、为什么改"

## 1. references/ 目录

- 路径：`/opt/alfq/references/`
- **gitignored**：不入 ALFQ 主仓库（许可证差异 + 体积）
- 仅供本地阅读和复制片段；**禁止直接 vendor 大段代码**
- 复制片段时必须：
  - 在文件头注释 `// Adapted from <project>/<path>@<commit>, <license>`
  - 检查许可证兼容性（见 §5）

## 2. 一键克隆脚本

`/opt/alfq/scripts/fetch-references.sh`（AI Agent 实施时按本说明创建）：

```bash
#!/usr/bin/env bash
# 克隆所有参考项目到 references/，已存在则 fetch 更新
# 用法: bash scripts/fetch-references.sh [project_name]
set -euo pipefail

REF_DIR="$(cd "$(dirname "$0")/.." && pwd)/references"
mkdir -p "$REF_DIR"

declare -A REPOS=(
  [bbgo]="https://github.com/c9s/bbgo.git"
  [nautilus_trader]="https://github.com/nautechsystems/nautilus_trader.git"
  [qlib]="https://github.com/microsoft/qlib.git"
  [gocryptotrader]="https://github.com/thrasher-corp/gocryptotrader.git"
  [Lean]="https://github.com/QuantConnect/Lean.git"
  [StockSharp]="https://github.com/StockSharp/StockSharp.git"
  [hftbacktest]="https://github.com/nkaz001/hftbacktest.git"
  [vectorbt]="https://github.com/polakowo/vectorbt.git"
  [quantstats]="https://github.com/ranaroussi/quantstats.git"
  [jesse]="https://github.com/jesse-ai/jesse.git"
  [barter]="https://github.com/barter-rs/barter.git"
  [zipline]="https://github.com/quantopian/zipline.git"
  [alphalens]="https://github.com/quantopian/alphalens.git"
  [pyfolio]="https://github.com/quantopian/pyfolio.git"
  [QuantDinger]="https://github.com/brokermr810/QuantDinger.git"
)

clone_one() {
  local name="$1"; local url="${REPOS[$name]}"
  local dir="$REF_DIR/$name"
  if [[ -d "$dir/.git" ]]; then
    echo "==> updating $name"
    git -C "$dir" fetch --depth=1 origin
  else
    echo "==> cloning $name"
    git clone --depth=1 "$url" "$dir"
  fi
}

if [[ $# -eq 0 ]]; then
  for k in "${!REPOS[@]}"; do clone_one "$k"; done
else
  clone_one "$1"
fi
```

## 4. 模块映射的硬性要求

每个 ALFQ 模块的 README 顶部必须有 `## References` 段，例如：

```markdown
# OMS - 订单管理

## References
- nautilus_trader@a1b2c3d: `nautilus_trader/execution/engine.pyx:120-450` (借鉴：状态机切换 + 重试退避)
- Lean@e4f5g6h:  `Engine/TransactionHandlers/BrokerageTransactionHandler.cs:200-380` (借鉴：对账流程)
- bbgo@i7j8k9l: `pkg/exchange/binance/exchange.go:50-200` (借鉴：Adapter 接口签名)

## 与参考的差异（为什么改）
- 我们用 client_order_id (UUIDv7) 作幂等键，nautilus 用 ClientOrderId+VenueOrderId 二元组——简化
- 我们走 Connect RPC，所有跨服务调用统一；nautilus 用进程内 MessageBus——分布式需要
- ...
```

PR 检查清单必须勾选 "References 段已填写"。

## 7. "改"的边界（不能照抄的部分）

| 维度 | 不抄，自研 |
|---|---|
| Connect+SSE 协议层 | 开源大多数用 REST/WS，我们一致走 Connect |
| 多租户 + RLS | 开源单租户为主 |
| DSL+ONNX 生产模型 | 开源多让用户写 Python 进生产，我们禁止 |
| Vault 集成 | 自建 |
| 中文 i18n + Notebook 沙箱 | 自建 |
| 文件大小/复杂度上限 | 开源项目普遍超我们上限，借鉴设计但**结构必须更扁平更小** |

## 8. 反模式（不要犯）

- ❌ 直接复制大段 AGPL（bbgo）代码到本仓库
- ❌ 把开源项目作 git submodule（污染依赖）
- ❌ 只读 API 文档不读源码就开写——容易抄错语义（尤其 DSL 算子边界）
- ❌ 抄了之后 README 不写 Reference 段
- ❌ 看 1 个项目就动手——至少看 2 个对照交叉验证
- ❌ 把 references/ 的代码当生产依赖（必须自己重写）

## 9. 工作流总结（AI Agent 每个模块必做）

```
[1] 读 docs/0X-*.md（本模块规范）
[2] bash scripts/fetch-references.sh <project>
[3] 按 §6 阅读路线读源码（每个文件做笔记到 docs/notes/<module>.md）
[4] 写 ADR（如果设计有偏离）
[5] 先定义 interface（在 domain/ 或 types.go）
[6] 先写测试 + golden 数据
[7] 实现 + 在 README 标注 Reference
[8] CI 全绿（含本文件大小复杂度检查）
[9] PR 描述写"借鉴了什么、改了什么、为什么"
```
