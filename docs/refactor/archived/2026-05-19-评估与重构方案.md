### 4.2 关键设计决策

#### 4.2.1 Bootstrap 模式（PR-3 核心）

**问题**：4 个 main.go 共 663 行，重复 80%。

**方案**：引入 `backend/go/internal/common/bootstrap` 包，采用 **Functional Options + Registrar** 模式。

```
internal/common/bootstrap/
├── bootstrap.go     # Run(svcName, register, opts...) error
├── deps.go          # type Deps struct { Cfg, Log, PG, RDB, NATS, CH }
├── server.go        # h2c server + graceful shutdown
├── healthz.go       # /healthz, /readyz, /metrics 默认实现
├── options.go       # WithoutPG(), WithoutRedis(), ...
├── *_test.go        # 表驱动 + httptest，覆盖率 ≥ 80%
└── README.md
```

**主调用方**：
```go
// cmd/trading-core/main.go (目标 ≤ 50 行)
func main() {
    if err := bootstrap.Run("trading-core", register); err != nil {
        log.Fatalf("bootstrap: %v", err)
    }
}

func register(mux *http.ServeMux, d *bootstrap.Deps) error {
    adp := adminapi.NewAdapter(d.PG, d.RDB, d.Log)
    p, h := alfqv1connect.NewAuthServiceHandler(adp); mux.Handle(p, h)
    p, h = alfqv1connect.NewStrategyServiceHandler(adp); mux.Handle(p, h)
    // ...
    return nil
}
```

**md-gateway / quant-engine 不需要 PG/Redis** → 用 options 关闭：
```go
bootstrap.Run("md-gateway", register, bootstrap.WithoutPG(), bootstrap.WithoutRedis())
```

#### 4.2.2 共享 Builder（PR-2 核心）

**问题**：4 个 Dockerfile 仅 `./cmd/<name>` 一处不同。

**方案**：单一 `backend/go/Dockerfile.builder`，通过 `--build-arg SVC=trading-core` 参数化。

```yaml
# docker-compose.prod.yml
trading-core:
  build:
    context: ..
    dockerfile: backend/go/Dockerfile.builder
    args: { SVC: trading-core }
```

#### 4.2.3 adminapi 拆分（PR-4 核心）

**问题**：`service.go` 330 行混合 5 个 service。

**方案**：按服务拆分，每文件单一 service，配套 `_test.go`。

```
internal/adminapi/
├── service.go           # ≤ 50 行：仅 Adapter 构造
├── auth_handler.go      # 已存在 221 行（不动）
├── strategy_handler.go  # 新建（从 service.go 抽出）
├── account_handler.go   # 新建
├── broker_handler.go    # 新建
├── backtest_handler.go  # 新建
├── audit_handler.go     # 新建
├── adapter.go           # 已存在，瘦身为 proto↔domain 转换
└── *_test.go            # 每 handler 一份
```

### 4.3 设计模式对照

| 模式 | 应用点 | 收益 |
|---|---|---|
| **Bootstrap / Composition Root** | `bootstrap.Run` | 统一启动流程，业务侧只关心 handler 注册 |
| **Functional Options** | `bootstrap.WithoutPG()` | 服务声明依赖，bootstrap 按需建立 |
| **Adapter Pattern** | `adminapi/adapter.go` | proto ↔ domain 边界清晰 |
| **Multi-stage Build + Build Arg** | `Dockerfile.builder` | 4 镜像共享构建缓存层 |
| **Single Responsibility** | adminapi 拆分 | 每文件一个 service |

### 4.4 性能瓶颈处理

> 现阶段**无明确性能瓶颈**（前端联调阶段）。本次重构**不引入性能优化**，避免与功能行为耦合。已识别但**延后处理**的项：

|---|---|---|

| ConnectRPC h2c 并发 | 默认 | 同上 |

### 6.2 回滚预案

每个 PR 设计为**独立可回滚**：

| 触发条件 | 操作 |
|---|---|
| 任一容器超过 60s 未 Up | `git revert HEAD` + `docker compose up -d --build` |
| 登录 5xx 重现 | 同上，并 grep `auth_handler.go` 修改 |
| 测试覆盖率回退 ≥ 5pp | 同上 |
| `pnpm build` bundle ≥ 110% 基线 | 仅回滚前端 PR |

### 6.3 不可逆操作（**本次重构禁止；如需启动，按附录 C 独立授权 PR**）

| 操作 | 本次重构 | 备注 |
|---|---|---|
| `git filter-branch` / `git filter-repo` 历史重写 | ❌ 禁止 | 缩减 `.git` 历史 pack 仅能通过此手段；启动需附录 C 流程 |
| `git push --force` / `--force-with-lease` | ❌ 禁止 | 同上；会使所有协作者本地 clone 失效 |
| 删除 `docs/*` | ⚠️ 需评估人确认 | 一般不允许 |
| 修改 `migrations/*.sql` | ❌ 范围外 | 数据安全 |

**设计取舍说明**：

`.git` 当前 64MB 中，~54MB 来自 commit `a0fbd50` 之前误提交的二进制，已被 git GC 合入历史 pack。在**不重写历史**的前提下，无论后续怎么 `git rm` 都**无法**让 `.git` 回到 ~10MB 量级——这是 git 内容寻址存储的固有特性。本方案据此做出取舍：

- ✅ **本次目标**：工作树二进制清零（PR-1），并通过 `.gitignore` + pre-commit hook 让 `.git` **停止增长**
- ❌ **本次不做**：历史 pack 缩减（属于不可逆操作，影响所有 clone）
- 📋 **未来可选**：按附录 C 流程发起独立授权 PR，由人工评估后执行

```yaml
          PCT=$(cd backend/go && go tool cover -func=cover.out | tail -1 | awk '{print $3}' | sed 's/%//')
          test "$(echo "$PCT >= 50" | bc)" -eq 1 || { echo "Coverage $PCT% < 50%"; exit 1; }
  golangci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: golangci/golangci-lint-action@v6
  frontend:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - run: cd frontend && pnpm install --frozen-lockfile && pnpm lint && pnpm build
```

### C.2 标准操作流程

```bash
# 1. 备份
git clone --mirror git@github.com:myfxlogs/AlfQ.git AlfQ.git.bak

# 2. 安装 git-filter-repo（推荐，不用已弃用的 filter-branch）
pip install git-filter-repo

# 3. 重写历史，移除指定大文件
cd AlfQ
git filter-repo --invert-paths \
  --path backend/go/trading-core \
  --path backend/go/quant-engine \
  --path frontend/package-lock.json \
  --path frontend/tsconfig.tsbuildinfo

# 4. 验证
du -sh .git                          # 应显著下降
git log --oneline | wc -l            # commit 数量应保持不变

# 5. 推送（破坏性！）
git remote add origin git@github.com:myfxlogs/AlfQ.git
git push --force --all
git push --force --tags

# 6. 通知所有协作者重新 clone
```
