# 17 - 发布与变更管理

> 企业级量化系统必须做到：**所有变更可追溯、可回滚、可控速**。本文规定 Feature Flag、灰度、变更评审、回滚 SOP、值班轮换。

## 1. 总体原则

1. **小批量、高频次**：每天可多次发布；不要憋大版本
2. **灰度优先**：默认按 tenant / account / 用户百分比灰度
3. **可回滚 > 修复前进**：出问题先回滚，再分析
4. **变更窗口**：实盘相关变更避免在亚盘/欧盘/纽盘开盘前 30 分钟与收盘前 30 分钟
5. **双人审批**：实盘 / 风控 / 资金路径变更必须两人 review

## 2. 版本与发布

### 2.1 版本号

`vMAJOR.MINOR.PATCH[-prerelease]`

| 字段 | 何时升 |
|---|---|
| MAJOR | proto 破坏性变更 / 数据迁移不可回滚 |
| MINOR | 向后兼容新特性 |
| PATCH | bug fix / 文档 / 内部重构 |
| prerelease | `-rc.1`, `-beta.1` |

每个服务、proto 包、前端、SDK 各自独立版本。

### 2.2 制品

- 容器镜像：`registry.alfq.io/<svc>:<version>` + `:<git_sha>` + `:latest`（仅 dev）
- 签名：cosign 签名 + SBOM（SPDX）
- 不可变：制品一旦推送，**禁止覆盖**；修补走新版本

### 2.3 环境流水

```
dev (每次 push)
  ↓ 自动
paper (PR 合入 main 自动部署)
  ↓ 手工触发 + 灰度
prod-canary (5% 流量 30min~24h)
  ↓ 自动晋升（或回滚）
prod-rolling (10% → 50% → 100%)
```

## 3. Feature Flag 平台

### 3.1 实现

- 第一阶段：**自研轻量 flag**（PG `feature_flags` 表 + Redis 缓存 + SSE 推送到所有服务）
- 第二阶段：考虑 OpenFeature + GrowthBook 集成

### 3.2 数据模型

```sql
CREATE TABLE feature_flags (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  key         TEXT NOT NULL UNIQUE,         -- e.g. "strategy.onnx_v2"
  description TEXT,
  type        TEXT NOT NULL,                -- bool / int / string / json
  default_val JSONB NOT NULL,
  rollout     JSONB NOT NULL,               -- 见下
  owner       TEXT,
  expires_at  TIMESTAMPTZ,                  -- 强制过期
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

`rollout` JSON 示例：

```json
{
  "rules": [
    {"if": {"tenant_id": ["t1","t2"]}, "value": true},
    {"if": {"env": "paper"}, "value": true},
    {"percent": 10, "salt": "strategy.onnx_v2", "value": true}
  ]
}
```

### 3.3 客户端

```go
flag := flags.Bool(ctx, "strategy.onnx_v2", false)
if flag {
    // 新路径
}
```

- ctx 注入 `tenant_id` / `user_id` / `env`，自动参与规则计算
- 缓存 30s + SSE 主动推送变更
- 评估时间 < 100µs

### 3.4 Flag 治理

- 每个 flag 必须有 `owner` 与 `expires_at`
- 到期未清理 → 周报警告 → 一个月内必须移除（永久 ON 或永久 OFF）
- 严禁在生产长期保留 flag

### 3.5 Flag 类别

| 类别 | 用途 | 示例 |
|---|---|---|
| Release | 灰度新功能 | `factor.dsl_v2` |
| Ops | 应急开关 | `oms.disable_market_order` |
| Permission | 试运行权限 | `ui.new_dashboard_enabled` |
| Experiment | A/B 实验（v2） | — |

## 4. 灰度发布策略

### 4.1 灰度维度（按优先级）

1. **环境**：dev → paper → prod
2. **租户白名单**：先开内部租户 + 1-2 个早期客户
3. **账户类型**：先 demo / paper，后 live
4. **流量百分比**：5% → 10% → 25% → 50% → 100%
5. **品种白名单**：先单一品种（如 EURUSD），后扩展

### 4.2 时间线模板

| 阶段 | 流量 | 持续 | 通过条件 |
|---|---|---|---|
| Canary | 5% | 30min – 24h | err < baseline × 1.1，p99 < baseline × 1.2 |
| Rollout-1 | 25% | 2h | 同上 |
| Rollout-2 | 50% | 4h | 同上 |
| GA | 100% | — | 24h 内无 P0/P1 |

### 4.3 自动晋升 / 回滚

- 监控 SLO 看板（详见 15 章），错误预算消耗 > 20% 自动暂停灰度
- 任何 P0 告警 → 自动 `docker compose up -d <svc>` 回滚镜像 tag / Flag OFF
- 脱机在 staging 复现、修复后重发

## 5. 数据库迁移管理

### 5.1 原则

- 迁移**只前进**（goose UP），回滚走数据修复脚本
- **分两步法**：先加列/索引（兼容旧版），后改代码引用，再清旧列
- 大表 DDL（ALTER）必须**在线 migration**：`pg_repack` / `pt-online-schema-change` 或分批
- ClickHouse：用 `ALTER TABLE` + 物化视图，避免锁

### 5.2 迁移流程

1. PR 包含 `migrations/postgres/NNN_xxx.sql`（up + down，down 可只是注释说明）
2. CI 在 testcontainers 上跑迁移 + 回滚（验证 down）
3. 部署前先在 paper 跑一次 + 性能评估
4. 生产：先迁数据库再发服务；服务必须**同时兼容新旧 schema**
5. 完成后**下个版本**再清旧 schema

### 5.3 回滚 SQL

每次迁移 PR 必附 `rollback.sql`，描述：
- 如何恢复（即使是手工恢复也写清楚）
- 数据丢失风险评估

## 6. 变更评审（Change Review）

### 6.1 分级

| 等级 | 触发条件 | 评审人 |
|---|---|---|
| L1 文档/UI 微调 | 不动业务逻辑 | 1 名 reviewer |
| L2 功能新增 | 非交易/风控/资金 | 2 名 reviewer |
| L3 交易/风控/资金 | 影响订单或风控 | 2 名 + risk_officer |
| L4 架构/proto 破坏性 | 跨服务 | 2 名 + 架构 owner + ADR |
| L5 紧急修复 | P0 故障 | 1 名 + on-call lead，事后补 ADR |

### 6.2 PR Checklist（强制）

- [ ] 测试通过 + 覆盖率不降
- [ ] 文档已更新
- [ ] Feature Flag（若有）已记录 owner / expires_at
- [ ] 监控/告警变更（若有）
- [ ] 数据迁移已附 rollback
- [ ] 风险评估（L3+）
- [ ] 回滚步骤
- [ ] 参考来源已标注（13 章要求）

### 6.3 紧急修复（hotfix）

- 走 `hotfix/<issue>` 分支，直接从生产 tag 分叉
- 最少 1 名 reviewer + on-call lead 批
- 24h 内必须补完整 PR 评审与文档

## 7. 回滚 Playbook

### 7.1 回滚优先级

1. **Feature Flag OFF**（秒级，零部署）
2. **回滚镜像 tag**：`IMAGE_TAG=<old> docker compose -f docker-compose.prod.yml up -d <svc>`（分钟级）
3. **数据库 down 迁移 / 数据修复脚本**（小时级，最后手段）

### 7.2 标准回滚命令

```bash
# Flag 回退
make flag-off KEY=strategy.onnx_v2

# 服务版本回退
IMAGE_TAG=<prev> docker compose -f /opt/alfq/deploy/docker-compose.prod.yml up -d trading-core

# 数据库（仅在 down 可逆时）
goose -dir migrations/postgres down 1
```

### 7.3 回滚记录

每次回滚必须：

1. 写 `docs/incidents/YYYY-MM-DD-<slug>.md`
2. 含：触发条件、影响范围、回滚动作、后续修复计划
3. 24h 内开 Postmortem 会议

## 8. 值班与 On-call

### 8.1 轮值

- 7×24 on-call 主备双人
- 主：1 周一轮；备：协助
- 节假日单独排班
- 工具：PagerDuty 或自建（结合 Telegram bot）

### 8.2 On-call 职责

- 接告警 5 分钟内确认（P0）/ 15 分钟（P1）
- 触发 Runbook（详见 11 章）
- 决定是否需要回滚 / KillSwitch
- 升级条件：30 分钟未恢复 → 升级到 lead

## 9. 发布日历与冻结期

### 9.1 冻结期

- 主要交易时段（伦敦/纽约开盘 ±30 分钟）禁止生产发布
- 大事件前后（非农、FOMC、年报季）禁止改风控/资金路径
- 节前最后一个工作日下午禁止发布

### 9.2 维护窗口

- 周末 UTC 02:00–04:00（forex 休市）为标准维护窗口
- 提前 48h 通告 + 在 Status Page 公示

## 10. Status Page

- 内部 status page（如 [Atlassian Statuspage](https://www.atlassian.com/software/statuspage) 或自建）
- 三类状态：Operational / Degraded / Outage
- 计划维护提前公告
- 客户可订阅邮件/Webhook

## 11. 监管/合规变更专项

- 任何影响审计、KYC、AML、数据保留的变更：
  - 走 L4 评审 + 法务/合规 sign-off
  - 不走 Feature Flag 灰度（一刀切，全部启用）
  - 变更前后审计日志 schema 必须明确
