# 14 - 领域模型与交易规则

> **本文件是金融领域的"事实层"**。所有计算口径必须以此为准；研究层、回测、实盘三处实现必须一致。任何对本文公式的偏离都会导致回测 vs 实盘不一致，**严禁 Agent 自行约定**。

## 1. 术语与基础概念

| 术语 | 定义 |
|---|---|
| **Bar / Candle** | 在固定时间窗口聚合的 OHLC + Volume |
| **Tick** | 单次 bid/ask 报价更新 |
| **Pip** | 报价最小变动单位。EURUSD 5 位报价时 1 pip = 0.0001；JPY 对 = 0.01 |
| **Point** | mtapi 中 `Point` 字段，定义在品种属性中（5 位报价时 1 point = 0.00001） |
| **Lot** | 标准手数。Forex 标准手 = 100,000 单位基础货币；mini = 10,000；micro = 1,000 |
| **Contract Size** | 1 lot 对应的合约规模，来自品种属性 |
| **Position** | 持仓，含方向、数量、加权均价、浮盈 |
| **Order** | 委托，可能是市价/限价/止损 |
| **Hedging Mode (MT4)** | 同品种允许多笔反向持仓，单独跟踪 |
| **Netting Mode (MT5)** | 同品种净持仓（多空相抵） |
| **Swap (Rollover)** | 隔夜利息，按持仓方向 + 品种 swap rate 计算 |
| **Spread** | ask - bid，单位 point |
| **Slippage** | 实际成交价与预期价的差 |
| **Margin** | 占用保证金 = (Volume × Contract Size × Price) / Leverage |
| **Free Margin** | Equity - Margin |
| **Margin Level** | Equity / Margin × 100% |
| **Stop Out Level** | 经纪商强平阈值（多为 50% 或 30%） |

## 2. 价格与精度

### 2.1 报价精度

- 每品种 `digits` 字段定义报价小数位（5/4/3/2）
- 5-digit broker（fractional pip）：EURUSD 报 1.12345，最小单位 0.00001 = 1 point
- **所有内部计算用 `Decimal`/`NUMERIC(20,8)`**，禁止 `float64` 直接比较价格
- 显示时按 `digits` 截断

### 2.2 Bid / Ask / Mid

- 买入用 **ask**（含点差），卖出用 **bid**
- 仓位浮盈计算：
  - Long: `(current_bid - open_price) × contract_size × volume`
  - Short: `(open_price - current_ask) × contract_size × volume`
- 中间价 mid = (bid + ask) / 2，**仅用于显示和指标计算，不用于撮合**

## 3. 订单生命周期细节

### 3.1 完整状态机

```
[NEW] ──validate──► [VALIDATED]
                       │
                       ├─reject─► [REJECTED]
                       │
                       └─risk_pass──► [RISK_APPROVED]
                                          │
                                          └─submit──► [SUBMITTED]
                                                         │
                                                ┌────────┼─────────────────┐
                                                ▼        ▼                 ▼
                                          [WORKING]  [PARTIALLY_FILLED]  [FILLED]
                                              │           │
                                              │           ▼
                                              │      [FILLED]
                                              │
                                              ├──cancel──► [CANCELLED]
                                              ├──expire──► [EXPIRED]
                                              └──error────► [FAILED]
```

每次状态变更必须：写 PG 行 + 发 NATS `order.evt.<account>.<id>` + 写 audit。

### 3.2 关键不变量

- `filled_qty <= qty`
- `filled_qty == qty` ⟺ `state == FILLED`
- `state in {CANCELLED, REJECTED, FAILED, FILLED, EXPIRED}` 为**终态**，不可再变
- 终态后到达的 broker 更新必须告警（异常对账场景）

### 3.3 幂等与重试

- `client_order_id` (UUIDv7) 唯一约束 `(account_id, client_order_id)`
- 重提相同 `client_order_id`：返回历史结果，不重复下单
- 网络失败重试：用同 `client_order_id` 调 broker `Query` 先确认状态再决定是否补发

### 3.4 MT4 vs MT5：协议级差异警示（重要 ⚠️）

> **MT4 与 MT5 是两套独立的协议与平台**，proto 定义、字段命名、枚举语义、撮合模型、对账接口均**不可共用**。
> 参考 proto 源：`/opt/alfq/gprc/mt4.proto`（1997 行）与 `/opt/alfq/gprc/mt5.proto`（2987 行），两份独立 schema。
> 实施时**禁止**把 MT4 的 message/enum 直接套用到 MT5（反之亦然），必须在 `md-gateway` / `oms` 中维护两套独立的 mapper。
>
> ⚠️ **`/opt/alfq/gprc/` 是 mtapi 官方目录，禁止新增或修改任何文件**（含 `mt4.proto` / `mt5.proto` / `mt4goExample/` / `mt5goExample/`）。Go 桩代码生成走 `backend/proto-mtapi/build.sh`（见 doc 25 §3）。
>
> **远端 gRPC 服务器**（gRPC over TLS，端口 443）：
>
> | 平台 | Endpoint |
> |---|---|
> | MT4 | `mt4grpc3.mtapi.io:443` |
> | MT5 | `mt5grpc3.mtapi.io:443` |

#### 协议结构差异

| 维度 | MT4 Hedging | MT5 Netting / Hedging |
|---|---|---|
| 持仓模型 | **Hedging**（同品种允许多笔反向单，每笔独立 ticket） | 默认 **Netting**（同品种净持仓），账号可选 Hedging |
| 持仓 ID | 每笔订单 = 一笔 ticket = 一条 position | Order / Deal / Position 三层分离 |
| 数据单元 | 单一 `Order`（含开仓+平仓+pending） | `Order` + `Deal`（成交回报）+ `OrderInternal` / `DealInternal` |
| 平仓接口 | `OrderClose` / `OrderCloseBy`（按 ticket） | `OrderClose`（反向下单减仓） |
| 部分平仓 | 仅 MT4 支持 `OrderCloseBy`（两单对冲平仓） | 无此接口 |

#### Proto 关键差异对照表

| 概念 | MT4 (`mt4.proto`) | MT5 (`mt5.proto`) | 备注 |
|---|---|---|---|
| 订单方向/类型 | `enum Op` (Buy/Sell/BuyLimit/.../Balance/Credit, 共 8 值) | `enum OrderType` (..+ BuyStopLimit/SellStopLimit/CloseBy, 共 11 值) | **MT5 多 StopLimit 与 CloseBy；不可枚举值映射** |
| 订单状态 | 无显式 state 字段（用 OpenTime/CloseTime 推断） | `enum OrderState` (Started/Placed/Cancelled/Partial/Filled/Rejected/Expired/...) | MT5 状态机更精细 |
| 成交回报 | 无独立 Deal 概念，靠 `OnOrderUpdate` 推差值 | `enum DealType` (DealBuy/DealSell/Balance/Credit/Charge/Correction/Bonus/Commission/...) | **MT5 把账户调整也作为 Deal** |
| 成交执行 | `enum Execution` (Request/Instant/Market) | `enum FillPolicy` (FillOrKill/ImmediateOrCancel/Return) | 完全不同语义 |
| 订单有效期 | `Expiration` 仅 timestamp | `enum ExpirationType` (GTC/Today/Specified/SpecifiedDay) + timestamp | MT5 显式枚举 |
| 账户类型 | `enum AccountType` | `enum AccMethod` | 命名不同 |
| 品种参数 | `enum ProfitMode` / `enum MarginMode` | `enum CalculationMode` / `enum GTCMode` | MT5 合并到 CalculationMode |
| 历史行情 | `QuoteHistory` (单个) / `QuoteHistoryMany` | `PriceHistoryEx` / `PriceHistoryExMany` / `PriceHistoryMonthMany` / `PriceHistoryTodayMany` | **MT5 接口更细分** |
| 实时事件流 | `OnOrderUpdate` / `OnQuote` / `OnTickValue` / `OnOrderProfit` (4 路) | 上述 + `Events`（统一事件流）+ `OnMarketWatch` / `OnTickHistory` / `OnMail` / `OnOpenedOrdersTickets` (9 路) | **MT5 流式接口数量翻倍** |
| 交易接口 | `OrderSend` / `OrderModify` / `OrderClose` / `OrderCloseBy` / `OrderDelete` (5 个) | `OrderSend` / `OrderModify` / `OrderClose` (3 个) | MT5 通过 OrderType + state 隐式覆盖 Delete |
| 服务划分 | `Connection` / `MT4` / `Service` / `Subscriptions` / `Trading` / `Streams` | 上述 + `QuoteHistory` / `TickHistory` | MT5 把历史数据拆为独立 service |

#### OMS 抽象层设计原则

1. **领域模型层**对上游屏蔽差异：
   - 统一 domain 概念：`Order` / `Position` / `Fill`（不要直接映射 MT4 `Order` 或 MT5 `Order/Deal`）
   - 上层只看"按 symbol 净持仓"视图 + 可选"按 ticket/position_id 明细"视图
   - 平台识别：`brokers.platform ∈ {mt4, mt5}`（参见 doc 02 `brokers` 表）
   - 持仓模式：建议 `accounts` 表新增 `position_mode TEXT NOT NULL DEFAULT 'auto'`（值：`hedging` / `netting` / `auto`，`auto` 由 platform 推导：mt4→hedging、mt5→netting；用户可显式覆盖以支持 MT5 Hedging 账号）
2. **Gateway 层**维护两套独立 mapper：
   - `internal/broker/mt4/`：包装 `mt4.proto` → 领域模型
   - `internal/broker/mt5/`：包装 `mt5.proto` → 领域模型
   - 共享接口 `BrokerGateway`（定义在 `internal/broker/types.go`），但**不共享 proto 类型**
3. **回测引擎**：需根据账号 `position_mode` 选择 Hedging / Netting 撮合逻辑（同品种反向单是否相抵）。
4. **DealType 特殊处理**（仅 MT5）：`Balance / Credit / Charge / Correction / Bonus / Commission / InterestRate` 等账户调整类型不进入 `orders` 表，应直接写入 `account_balance_snapshots` + `audit_log`。
5. **CloseBy 接口**（仅 MT4）：OMS 内部可建模为"两笔反向 ticket 同时 close"，不必为 MT5 复用此 RPC。

### 3.5 部分成交合并

mtapi 不直接暴露分笔成交，但同 ticket 多次成交价不同时，OMS 按以下规则维护：

```
avg_fill_price = Σ(fill_qty_i × fill_price_i) / Σ(fill_qty_i)
filled_qty     = Σ(fill_qty_i)
```

每次 broker 推送的 `OnOrderUpdate` 若 `OrderTotalVolume > 上次记录`，差值即为新成交。

## 4. 撮合 / 滑点 / 手续费 / Swap 模型

### 4.1 撮合规则（回测引擎必须实现，对齐实盘）

| 订单类型 | 实盘撮合 | 回测撮合 |
|---|---|---|
| **市价单** | 立即按当前 ask/bid 成交 + 滑点 | 用 next tick 对侧价 + 滑点 |
| **限价单** | 触达限价方向时成交 | 触达即成交（默认）；可配置成交概率 |
| **止损单** | 触发后转市价 | 同上 |
| **GTC / GTD** | 按经纪商规则 | 模拟到期取消 |

### 4.2 滑点模型

公式：

```
slippage_points = max(0, normal(μ_slip, σ_slip))    # 默认 μ=0 σ=spread*0.3
fill_price_long  = ask + slippage_points × point
fill_price_short = bid - slippage_points × point
```

参数可按品种/时段配置（伦敦/纽约/亚盘流动性差异）。

### 4.3 手续费

```
commission = commission_per_lot × volume × (round_trip ? 2 : 1)
```

- 默认按 `accounts.commission_per_lot` 配置（USD/lot）
- 部分品种采用按金额 bps：`commission = notional × bps / 10000`
- 由 `brokers` 表的 `commission_model` 字段控制

### 4.4 Swap（隔夜利息）

```
swap_charge_long  = swap_long_points  × point × contract_size × volume × (周三 3倍 else 1)
swap_charge_short = swap_short_points × point × contract_size × volume × (周三 3倍 else 1)
```

- 每个经纪商每日 server 时间 22:00（或 broker 配置）计提
- 周三计 3 倍（覆盖周末）
- swap rates 从 `symbols.meta.swap_long/swap_short` 同步（mtapi `SymbolParams` 提供）

### 4.5 保证金

```
margin_required = (volume × contract_size × open_price) / leverage × margin_rate
```

`margin_rate`：多数品种 1，部分（如某些指数）有 1.5x；`leverage` 来自账户。

### 4.6 隔夜浮盈 / 已实现 PnL

- **未平仓持仓**：每 tick 更新浮盈，每日 server 00:00 归零（不归 cash，仅显示重置）
- **平仓**：浮盈结转为已实现 PnL，写入 `fills` + `account_balance_snapshots`

## 5. 交易日历与会话

### 5.1 时区基准

| 时间 | 时区 | 用途 |
|---|---|---|
| 系统 / 数据库 / 日志 | **UTC** | 内部唯一基准 |
| 用户 UI 显示 | 用户偏好 / 浏览器 | 仅展示 |
| MT 服务器时间 | broker 配置（多为 GMT+2/GMT+3 + DST） | 计算 swap 时间点 |
| 品种 session | 品种属性（mtapi `SymbolSessionTrade`） | 决定是否可下单 |

### 5.2 交易日历表

新增 `trading_calendar` 表（v1 简化版）：

```sql
CREATE TABLE trading_calendar (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  broker_id    UUID NOT NULL REFERENCES brokers(id),
  symbol       TEXT,                       -- NULL = 全品种
  weekday      SMALLINT,                   -- 0=Sun..6=Sat, NULL=单日特例
  date         DATE,                       -- 节假日特例
  open_utc     TIME,                       -- session 开盘 UTC
  close_utc    TIME,                       -- session 收盘 UTC
  kind         TEXT NOT NULL,              -- regular / holiday / early_close
  note         TEXT
);
```

- 默认从 mtapi `SymbolParams` 拉取并转换为 UTC 存储
- 节假日通过定期手动维护或第三方 ICS 同步
- 风控规则 `session.allow_only_in_session`：每笔下单前检查时段

### 5.3 周末 / 节假日 / 早收盘

- 周末（Sat/Sun）：默认 forex 关闭，加密 24/7
- 节假日：`kind=holiday`，整日禁交易
- 早收盘：`kind=early_close`，覆盖默认时段
- 夏令时：`open_utc` 已转 UTC，DST 切换日由后台脚本重写一次

### 5.4 实施约束

- 信号生成时即检查 session：若不在交易时段且策略未明示 GTC，则**缓存信号到下次开盘**或**丢弃**（按 spec 配置）
- 风控前置：拒绝非交易时段订单（除非显式 `force_after_hours`）

## 6. 资金分配与风险预算（多策略共享账户）

### 6.1 资金分配模型

`strategy_deployments.capital_alloc` 字段三种语义：

| 模式 | 含义 | 计算 |
|---|---|---|
| `fixed_cash` | 固定金额 | 直接使用 |
| `pct_equity` | 账户净值百分比 | `equity × pct` |
| `risk_parity` | 按波动率倒数加权 | 后台周期性调整 |

### 6.2 仓位计算（spec 中 position_sizing）

| type | 公式 |
|---|---|
| `fixed_lot` | `volume = lot` |
| `fixed_cash` | `volume = cash / (price × contract_size)` |
| `fixed_risk` | `volume = (capital_alloc × risk_pct) / (sl_distance × point_value)` |
| `kelly` | `f* = (bp − q) / b`，限制最大 `kelly_cap`（如 0.25） |
| `vol_target` | `volume = target_vol / realized_vol × scale` |

### 6.3 风险预算

每策略部署在 `params` 中声明：

```json
{
  "risk_budget": {
    "max_daily_loss": 200,             // USD
    "max_drawdown_pct": 0.05,          // 5%
    "max_position_notional": 50000,
    "max_correlation_with_existing": 0.7
  }
}
```

`trading-core` 在订单前检查：
1. 当日已实现 PnL + 未实现 PnL < 阈值
2. 跨策略相关性（若开启）：用近 N 日收益序列计算两两 corr
3. 总杠杆不超过账户 leverage × 0.8

### 6.4 资金冲突处理

多策略对同账户并发下单：
- OMS 用账户级互斥锁（Redis NX 100ms）串行化
- 每次下单前重读 free_margin
- margin 不足 → 拒单 + 写 risk_event(warn)

## 7. 浮动 PnL 与对账

### 7.1 浮动 PnL 计算

```
floating_pnl = Σ_positions (
    (current_bid - open_price) × contract_size × volume_long
  + (open_price - current_ask) × contract_size × volume_short
  - swap_accumulated
  - commission
)
equity = balance + floating_pnl
```

每秒重算 / 每 tick 重算（实时仪表盘）。

### 7.2 对账（每分钟）

```
broker_state  = mtapi.OpenedOrders + mtapi.AccountSummary
local_state   = PG positions + accounts.balance
diff = compare()
if diff.has_difference:
    log warn + risk_event
    if diff.persistent_for > 5min:
        risk_event critical + auto-halt deployment + 告警 P1
```

对账维度：
- 持仓量 / 均价 / 浮盈
- 余额 / 净值 / 已用保证金
- 在途订单 ticket 集合

### 7.3 真理源

| 字段 | 真理源 | 处理冲突 |
|---|---|---|
| 持仓 / 余额 / 净值 | **broker** | 以 broker 为准重置本地 |
| 订单 client_order_id | **本地** | broker 不存 client_id 我们维护 |
| 策略归属 / 部署关系 | **本地** | broker 不知道 |
