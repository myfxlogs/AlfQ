> 项目代号：**ALFQ**  
> 适用场景：基于 MetaTrader 4 / 5（通过 mtapi gRPC 网关）的多账户、多策略、多租户量化交易平台

## 1. 项目目标与边界

### 1.1 目标

- 支撑 **多账户 / 多经纪商 / 多策略** 的 7×24 自动化交易（外汇为主，预留股票/加密扩展）
- 提供 **从因子研究 → 回测 → 模拟盘 → 实盘** 的完整闭环
- **可观测、可风控、可灰度**，满足企业级运维要求
- 单策略下单延迟（本地侧）**p99 < 5 ms**，端到端（含 broker）**p99 < 150 ms**

### 1.2 非目标（一期不做）

- 亚毫秒级 HFT / 做市（受 mtapi 网关延迟限制，不在能力范围内）
- 撮合自研（直接走经纪商）
- 面向 C 端的资管 / 跟单业务（仅自营）

### 1.3 资产与频率定位

| 维度 | 选型 |
|---|---|
| 资产 | 外汇（FX）、贵金属、指数 CFD |
| 频率 | tick → 1m → 1h 多频率混合，以 **分钟~小时 CTA / 多因子打分** 为主 |
| 策略类型 | 趋势、均值回归、统计套利、跨品种对冲、事件驱动 |

### 2.2 关键设计原则

- **研究与实盘解耦，但共享 schema**：用 Protobuf 定义信号、订单、持仓、行情，Python/Go 共用同一份 `.proto`
- **因子用 DSL 表达不用代码**：因子定义为字符串表达式（如 `"ema($close, 20) / ema($close, 60) - 1"`），Python/Go 各有一套解释器，**因子本身只写一次**（核心降本手段）
- **一切配置化**：策略参数、风控阈值、品种白名单全部 YAML，Git 管理，热加载
- **消息总线为骨干**：所有跨服务通信走 NATS JetStream（行情 fan-out、订单事件、风控指令）
- **幂等优先**：所有下单带 `client_order_id`，重试不产生重复单

## 5. 模块详细设计

### 5.1 md-gateway（行情网关，Go）

**职责**：消费 mtapi `Streams.OnQuote` / `OnTickValue`，规范化后 fan-out 到 NATS

- **入**：`mtapi.Streams.OnQuote(id)` → `OnQuoteReply` 流
- **出**：NATS subject `md.tick.<broker>.<symbol>` （Protobuf 编码）
- **关键能力**
  - 多连接池（每经纪商 N 个连接，按品种分片）
  - 自动重连（基于 `CheckConnect` RPC + 指数退避）
  - 心跳监控：超过 N 秒无 tick 触发告警与切换
  - tick 落盘到 ClickHouse（异步批量写入，背压保护）
- **参考**：`bbgo` 的 `Stream` 抽象 + `gocryptotrader` 的 `Websocket` 重连机制

### 5.2 factor-svc（因子服务，Go）

**职责**：基于行情流，**增量计算**因子值

- **因子 DSL**：抄 `qlib` 的 `Expression`
  - 原子算子：`$open / $high / $low / $close / $volume / $bid / $ask`
  - 函数：`ema(x, n)`, `sma(x, n)`, `std(x, n)`, `ref(x, n)`, `corr(x, y, n)`, `rank(x)` 等
  - 示例：`"ema($close, 20) / ema($close, 60) - 1"`
- **存储**：因子值落 ClickHouse 表 `factor_values(ts, symbol, factor_name, value)`
- **下游**：策略服务订阅 `factor.<name>.<symbol>` 主题
- **关键**：Python 端用同一份 DSL，**研究和实盘因子定义文件 100% 相同**

### 5.3 strategy-svc（策略服务，Go）

**职责**：消费因子 + 行情 → 产生 `Signal`

- **策略接口**（抄 bbgo）：
  ```go
  type Strategy interface {
      ID() string
      Subscribe(ctx context.Context) []Subscription
      OnTick(ctx context.Context, tick *Tick) ([]*Signal, error)
      OnFactor(ctx context.Context, f *FactorValue) ([]*Signal, error)
      OnBar(ctx context.Context, bar *Bar) ([]*Signal, error)
  }
  ```
- **加载方式**：插件化（Go plugin 或编译进二进制 + 配置开关）
- **配置热加载**：参数 YAML 改动自动 reload（fsnotify + 灰度切换）
- **多策略隔离**：每策略独立 goroutine + 独立资金桶

### 5.4 risk-svc（风控服务，Go）—— 最关键

**所有订单必须经过此服务**，规则引擎驱动：

- **盘前检查**
  - 单笔最大手数 / 名义本金
  - 单品种持仓上限
  - 品种白名单 / 黑名单
  - 当前是否在可交易时段
  - 保证金可用率
- **盘中监控**
  - 实时净值 / 浮亏 / 回撤
  - 单日 / 单周 / 单月止损线
  - 滑点异常（实际成交 vs 信号价偏差）
  - 心跳丢失熔断（订阅 `md-gateway` 心跳）
- **Kill Switch**：HTTP/CLI 一键调用 → 拒绝新单 + 平所有仓 + 断连
- **状态机**：参考 Lean 的 `BrokerageMessageHandler` 流程

### 5.5 oms（订单管理，Go）

- **订单生命周期状态机**（简表，**完整定义见 doc 14 §3.1**）：`new → validated → risk_approved → submitted → working/partially_filled/filled`，失败侧 `rejected / failed / cancelled / expired`
- **幂等键**：内部 `client_order_id`（UUID v7）→ broker `ticket`
- **重试策略**：
  - 网络错误：指数退避，最多 3 次
  - `TIMEOUT`（mtapi error `0x105`）：查询订单状态后决定补单/放弃
  - `REQUOTE`（138）/ `PRICE_CHANGED`：按策略重报
  - `INVALID_TOKEN`（`0x100`）/ `CONNECT_ERROR`（`0x104`）：触发重连流程
- **对账**：每分钟比对本地账本 vs `OpenedOrders` / `OrderHistory`，差异告警

### 5.6 admin-api（管理后台 BFF，Go）

- 账户管理、策略上下线、参数热改、风控阈值调整、Kill Switch
- 前端：React + shadcn/ui（或简单做 Grafana 大盘 + Swagger UI）

### 5.7 研究层（Python）

- **数据**：直读 ClickHouse（`clickhouse-connect`）+ Polars
- **因子**：与 Go 共用 DSL 字符串；Python 端用 `polars` 表达式编译执行
- **回测引擎**
  - **向量化**：抄 `vectorbt`，秒级跑参数网格
  - **事件驱动**：抄 `nautilus_trader` 简化版，逐 tick 撮合，验证执行细节
  - 两套结果对齐 < 1% 误差才允许策略上线
- **报告**：抄 `quantstats`，输出 HTML（Sharpe / Sortino / Calmar / MaxDD / 月度热力图）
- **优化**：`optuna` + walk-forward
- **产物**：策略参数 YAML 推送到 `configs/strategies/`，触发 CI → 灰度上线

- 模型文件（ONNX）
- 日志归档

## 7. 接口与协议约定

### 7.1 内部 Protobuf（`proto/alfq/v1/`）

核心消息草案：

```proto
// marketdata.proto
message Tick {
  string broker = 1;
  string symbol = 2;
  int64  ts_ns  = 3;
  double bid    = 4;
  double ask    = 5;
  uint64 seq    = 6;
}

// signal.proto
message Signal {
  string strategy_id = 1;
  string symbol      = 2;
  int64  ts_ns       = 3;
  enum Side { FLAT=0; LONG=1; SHORT=2; }
  Side   target_side = 4;
  double target_qty  = 5;   // 目标手数（绝对值）
  double limit_price = 6;   // 0 表示市价
  string client_id   = 7;
}

// order.proto
message OrderRequest {
  string client_order_id = 1;  // 幂等键
  string account_id      = 2;
  string symbol          = 3;
  enum Type { MARKET=0; LIMIT=1; STOP=2; }
  Type   type            = 4;
  double qty             = 5;
  double price           = 6;
  double sl              = 7;
  double tp              = 8;
}
```

### 7.2 NATS Subject 命名

```
md.tick.<broker>.<symbol>             # 行情
md.bar.<broker>.<symbol>.<period>     # K线
factor.<name>.<symbol>                # 因子
signal.<strategy>.<symbol>            # 信号
order.req                             # 订单请求（风控订阅）
order.evt.<account>                   # 订单事件
risk.cmd                              # 风控指令（kill-switch 等）
sys.heartbeat.<service>               # 心跳
```

## 8. 风控与熔断

### 8.1 三层风控

| 层 | 位置 | 检查项 | 时延要求 |
|---|---|---|---|
| L1 策略内 | strategy-svc | 业务逻辑层自检 | 即时 |
| L2 风控网关 | risk-svc | 仓位/资金/品种/时段/滑点 | < 1 ms |
| L3 经纪商 | mtapi | 保证金/市场状态 | broker 侧 |

### 8.2 熔断触发条件（示例）

```yaml
circuit_breakers:
  daily_loss_pct: 2.0          # 日亏损超 2% 暂停开仓
  max_drawdown_pct: 5.0        # 回撤超 5% Kill Switch
  reject_rate_5m: 0.3          # 5 分钟拒单率 > 30% 暂停该策略
  md_silence_seconds: 10       # 行情静默 10s 切换源
  slippage_zscore: 4.0         # 滑点异常分布
```

### 8.3 Kill Switch 流程

1. 拒绝所有新订单（`risk-svc` 状态置 `HALT`）
2. 撤销所有挂单（OMS 逐账户调 `OrderDelete`）
3. 按配置决定是否一键平仓（`OrderClose`）
4. 通知告警通道（Telegram / 钉钉 / PagerDuty）
5. 等待人工解锁

## 9. 可观测性（监控/日志/告警）

### 9.1 指标（Prometheus）

| 类别 | 关键指标 |
|---|---|
| 行情 | `md_tick_rate{symbol}`、`md_latency_seconds`、`md_gap_count` |
| 策略 | `strategy_signal_count`、`strategy_pnl`、`strategy_position` |
| 订单 | `order_submit_total{result}`、`order_latency_seconds`、`order_reject_total{reason}` |
| 风控 | `risk_reject_total{rule}`、`risk_breaker_state` |
| 系统 | `goroutine`、`go_gc_pause_seconds`、`process_resident_memory_bytes` |

### 9.2 日志（Loki）

结构化 JSON，必带字段：`trace_id`, `account`, `strategy`, `symbol`, `client_order_id`

### 9.3 链路追踪（OpenTelemetry）

下单全链路：`signal → risk → oms → mtapi → broker`，可视化 p50/p99

### 9.4 告警分级

| 级别 | 通道 | 例 |
|---|---|---|
| P0 | PagerDuty + 电话 | Kill Switch 触发、所有连接断开 |
| P1 | Telegram + 钉钉 | 单策略熔断、行情中断 |
| P2 | 邮件 | 拒单率上升、对账差异 |

## 10. 部署与运维

### 10.1 环境分层

- `dev`：本地 docker-compose
- `paper`：模拟盘账户，全功能压测
- `prod`：实盘，多区域 VPS（伦敦/纽约/东京，就近经纪商机房）

### 10.2 高可用

- 行情网关：双活，按品种分片，互为热备
- 策略服务：单活（避免重复下单），故障切换 < 10s
- 风控/OMS：主从 + 共享 Postgres
- mtapi 连接：心跳 + 自动 `CheckConnect` 重连

### 10.3 灾备

- 配置 Git 异地仓库
- 数据 ClickHouse 跨机房复制
- 关键秘钥 Vault 多副本

## 11. 研发流程与 CI/CD

### 11.1 分支

`main`（保护）→ `release/*` → `feat/*` / `fix/*`

### 11.2 流水线

```
push → lint(golangci-lint + ruff) → ut → build → 
    proto 一致性检查 → 回测基线对比 → docker build → 
    paper 环境部署 → 自动冒烟 → 人工审批 → prod 灰度（10%→50%→100%）
```

### 11.3 策略上线门槛

1. 回测 Sharpe > 1.0、MaxDD < 策略目标 1.5×
2. 向量化回测与事件驱动回测对齐误差 < 1%
3. paper 盘运行 ≥ 5 个交易日，无重大风控事件
4. 双人 code review + 风控负责人签字
