# 21 - 跨服务一致性与幂等

> 由于 PG / CH / Redis / NATS / mtapi 不共享事务，跨服务必须用 **Saga + Outbox + 幂等键**保障最终一致性。本文给出标准模式，AI Agent 实施所有跨服务流程必须套用。

## 1. 核心问题

订单流程示意：

```
Client → trading-core → trading-core.Check → oms.Submit → BrokerAdapter → mtapi
                                          │
                                          ├─ 写 PG orders
                                          ├─ 发 NATS order.evt
                                          └─ 写 CH fills
```

任何一步失败 → 数据不一致风险。

## 2. 一致性模式选择

| 场景 | 模式 |
|---|---|
| 多服务、最终一致 OK | **Saga（编排式）+ 补偿** |
| 单服务多 DB 写 | **Outbox 模式** |
| 同一服务幂等重试 | **Idempotency Key** |
| 跨服务读最新值 | **Read-Your-Writes via 主库** |
| 高频写一致性弱要求 | **Async fire-and-forget + 对账** |

## 3. 幂等键规范

### 3.1 客户端职责

- 所有写 RPC 必须带 `Idempotency-Key`（Connect header）
- 值为 **UUIDv7**（含时间序），16-36 字符
- 客户端在重试时使用**同一**键

### 3.2 服务端流程（trading-core / oms / quant-engine）

```
1. 取 Idempotency-Key（缺失 → ERR_IDEMPOTENCY_KEY_MISSING）
2. Redis SETNX lock:idem:<key> ttl=300s
   - 占锁失败 → 查 idempotency_records 表
       - 已完成 → 返回缓存结果
       - 进行中 → ERR_IDEMPOTENCY_CONFLICT（409）+ Retry-After
   - 占锁成功 → 继续
3. 执行业务
4. 写 idempotency_records(key, request_hash, response, status, expires_at)
5. 释放锁（业务完成与否都释放）
```

### 3.3 持久化表

```sql
CREATE TABLE idempotency_records (
  key             TEXT PRIMARY KEY,
  tenant_id       UUID NOT NULL,
  user_id         UUID,
  request_hash    TEXT NOT NULL,          -- sha256(method+body)
  response        JSONB,
  status          TEXT NOT NULL,          -- pending / done / failed
  http_status     INT,
  expires_at      TIMESTAMPTZ NOT NULL,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_idem_expires ON idempotency_records(expires_at);
```

- TTL 5 分钟
- request_hash 不匹配 → 同 key 不同请求 → 拒绝（403）

### 3.4 OMS 订单幂等

OMS 使用业务级 `client_order_id` 唯一约束：

```sql
UNIQUE (account_id, client_order_id)
```

重试同 `client_order_id` → 查询历史记录返回，不重复下单。

## 4. Saga 模式

### 4.1 编排式 Saga（推荐）

> **状态名以 doc 14 §3.1 为准**：`new → validated → risk_approved → submitted → working/partially_filled/filled`，失败侧 `rejected / failed / cancelled / expired`。

订单 Saga 步骤：

```
S1. 写 orders(state=new) + 入参校验
       校验失败 → orders(state=rejected, reject_reason="validation:...") END
S2. orders(state=validated) → 调 RiskCheckService.Check
       风控拒绝 → orders(state=rejected, reject_reason="risk:...") END
S3. orders(state=risk_approved)
S4. BrokerAdapter.Submit
       提交失败/超时 → orders(state=failed) END，释放保证金缓存
S5. orders(state=submitted, broker_ticket=...)
S6. 等待 broker 推 OnOrderUpdate（异步）→ working/partially_filled/filled/cancelled/expired
```

### 4.2 补偿（Compensation）

每步必须明确"如果后续失败，如何回滚":

| 步骤 | 失败动作 |
|---|---|
| S1 写 PG 失败 | 直接报错（业务事务回滚） |
| S2 校验失败 | state=rejected（validation） |
| S2 风控失败 | state=rejected（risk） |
| S4 broker 提交失败 | state=failed；释放保证金缓存 |
| S5 之后 broker 主动拒单 | state=rejected（broker_reject）；通知用户 |
| S6 长时间无回执 | 走对账（5 分钟） + 主动 Query；超时仍无 → state=failed |

### 4.3 Saga 状态管理

订单本身的 state 列即 Saga 状态。复杂多步 Saga 用独立 `sagas` 表：

```sql
CREATE TABLE sagas (
  id          UUID PRIMARY KEY,
  type        TEXT NOT NULL,         -- order_placement / strategy_deploy / ...
  state       TEXT NOT NULL,         -- pending / step_n / done / failed / compensated
  steps       JSONB NOT NULL,        -- [{name, status, started_at, ended_at, error}]
  context     JSONB,
  started_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  ended_at    TIMESTAMPTZ
);
```

## 5. Outbox 模式

### 5.1 问题

服务要"写 DB + 发 NATS"，两者非原子：
- 先写 DB 后发 NATS：NATS 失败 → 消息丢
- 先发 NATS 后写 DB：DB 失败 → 假消息

### 5.2 解决：Outbox 表

```sql
CREATE TABLE outbox (
  id          BIGSERIAL PRIMARY KEY,
  aggregate   TEXT NOT NULL,         -- e.g. order
  aggregate_id TEXT NOT NULL,
  topic       TEXT NOT NULL,         -- NATS subject
  payload     BYTEA NOT NULL,        -- protobuf 序列化
  headers     JSONB,                 -- trace_id, tenant_id, ...
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  published_at TIMESTAMPTZ,
  attempts    INT NOT NULL DEFAULT 0,
  next_retry_at TIMESTAMPTZ
);
CREATE INDEX idx_outbox_unpublished ON outbox(published_at) WHERE published_at IS NULL;
```

### 5.3 流程

1. 业务事务：在**同一 DB tx** 中写业务数据 + 写 outbox 行
2. 异步 publisher：扫 `published_at IS NULL`，发 NATS，成功后更新
3. 失败重试：`attempts++`，指数退避，超过 N 次告警

### 5.4 实现

`backend/go/internal/common/outbox/`：

- `tx.AddEvent(ctx, topic, msg)` 在事务中追加
- `Publisher` goroutine 周期扫描
- NATS 用 JetStream（持久化 + at-least-once）

### 5.5 消费者去重

NATS 消费者必须幂等：
- 消息含 `event_id`（uuid）
- 消费方用 Redis `SETNX seen:<event_id>` 或 DB 唯一约束去重
- 处理失败 NAK 重投

### 5.6 事件编码规范（**protobuf binary**）

所有 NATS JetStream 消息体一律使用 **protobuf binary**（不是 JSON），以保持跨服务契约的强类型与版本演进能力。

**Schema 单一源**：`backend/proto/alfq/v1/events.proto`（M1 落地），定义：

```proto
// 通用 envelope，每个 NATS subject 的 payload 都包成这个
message EventEnvelope {
  string  event_id    = 1;  // uuidv7，去重键
  string  trace_id    = 2;  // OpenTelemetry trace
  string  tenant_id   = 3;
  string  aggregate   = 4;  // order / position / tick / risk_event ...
  string  aggregate_id = 5;
  google.protobuf.Timestamp occurred_at = 6;
  uint32  schema_version = 7; // 与 body 配套，向前兼容
  bytes   body        = 8;  // 子事件 protobuf 序列化（如 OrderEvent）
}

// 子事件示例（独立 message，不要复用 RPC request/reply）
message OrderEvent { ... }
message FillEvent { ... }
message PositionEvent { ... }
message TickEvent { ... }
message RiskEvent { ... }
```

**编码规则**：
- 发布方：`payload, _ := proto.Marshal(envelope)` 写 `outbox.payload BYTEA`，publisher 直接 `nc.Publish(subject, payload)`
- 消费方：`proto.Unmarshal(msg.Data, &envelope)`，再按 `aggregate` 分发到具体 handler
- NATS header 仅用于路由元数据（`Nats-Msg-Id` = `event_id` 用于服务端去重），业务字段一律放 envelope

**禁止**：
- ❌ NATS payload 用 JSON（破坏跨服务类型契约）
- ❌ 直接复用 RPC `Request`/`Reply` 作为事件 message（事件是独立生命周期）
- ❌ 跨 broker 平台共用事件 message（参考 doc 14 §3.4，MT4 / MT5 各自独立 event）

**版本演进**：
- 字段只增不删；删除走 `reserved`
- 不兼容变更 → 升 `schema_version` 并新建 subject（如 `order.evt.v2.<account>.<id>`）
- 双跑期间两个 consumer 并行消费

## 6. 顺序保证

| 场景 | 保证 |
|---|---|
| 同 account 订单事件 | NATS subject 按 `order.evt.<account>.<id>` 分区；同 account 单 consumer instance |
| 同 symbol tick | md-gateway 单连接顺序产出；下游不要求 |
| 跨 account | 不保证全局顺序，按时间戳 |

JetStream 配置：subject 用 `OrderedConsumer` 或 `MaxAckPending=1` 实现 per-key 顺序。

## 7. 对账机制

| 维度 | 频率 | 工具 |
|---|---|---|
| OMS 订单 vs broker | 每分钟 | oms reconciler |
| 资金 / 持仓 vs broker | 每分钟 | 同上 |
| outbox 未发布累积 | 每 5 秒 | publisher self-monitor |
| idempotency 表清理 | 每小时 | janitor job |
| sagas 卡住 5 分钟以上 | 每分钟 | saga monitor |

差异 → risk_event(warn) → 持续 → critical → 自动 halt + 告警。

## 8. 死信处理

- NATS DLQ：消费失败 N 次 → 转 `dlq.<original_subject>`
- 每日 Ops 检查 DLQ；提供工具 `make replay-dlq SUBJECT=...`
- 重要 DLQ 报警

## 9. 跨语言事务（不存在）

- **禁止**期望 PG → CH 在一个事务里
- CH 写入失败 → 业务表标记 `ch_sync_pending`，后台 worker 重试
- 极端情况下 CH 数据可能滞后 PG 几秒，前端展示注明"实时数据"vs"分析数据"

## 10. 实例：策略部署 Saga

```
DeployStrategy(strategy_id, account_id, params):
  S1. 校验权限 + spec → 创建 strategy_deployments(state=pending)
  S2. 下载 ONNX 模型到 quant-engine 本地 cache
       失败 → state=failed END
  S3. 校验模型 hash + signature
       失败 → state=failed END
  S4. 通知 quant-engine 加载部署
       超时/失败 → state=failed，清理 cache
  S5. quant-engine 订阅因子流，回执
  S6. state=running
  
KillSwitch(deployment_id):
  C1. state=halting
  C2. 发 NATS risk.cmd.halt
  C3. 等待 quant-engine 回执（5s 超时）
  C4. 平所有持仓（可选）
  C5. state=halted
```
