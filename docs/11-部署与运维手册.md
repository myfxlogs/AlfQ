# 11 - 部署与运维手册

## 1. 环境

| 环境 | 用途 | 形态 |
|---|---|---|
| **dev** | 本地开发 | docker-compose |
| **paper** | 联调 / 模拟盘 | 单机 K3s 或 compose |
| **prod** | 生产 | K8s + Helm + ArgoCD |

## 2. 开发环境（docker-compose）

`deploy/compose/docker-compose.yml` 至少包含：

```yaml
services:
  postgres:    { image: postgres:16, env: ..., volumes: pg-data }
  clickhouse:  { image: clickhouse/clickhouse-server:24, volumes: ch-data }
  redis:       { image: redis:7-alpine }
  nats:        { image: nats:2.10, command: -js }
  minio:       { image: minio/minio, command: server /data }
  vault:       { image: hashicorp/vault, command: server -dev }       # dev 用 -dev
  prometheus:  { image: prom/prometheus, volumes: cfg }
  grafana:     { image: grafana/grafana, volumes: data }
  loki:        { image: grafana/loki }
  tempo:       { image: grafana/tempo }
  jaeger:      { image: jaegertracing/all-in-one }
  jupyterhub:  { build: ./notebook }
  admin-api:   { build: ../../backend/go/cmd/admin-api }
  md-gateway:  { build: ../../backend/go/cmd/md-gateway }
  factor-svc:  { ... }
  strategy-svc:{ ... }
  risk-svc:    { ... }
  oms:         { ... }
  web:         { build: ../../frontend }
```

启动：

```bash
make dev-up         # docker compose -f deploy/compose/docker-compose.yml up -d
make dev-migrate    # 跑 PG/CH 迁移
make dev-seed       # 种子数据（默认 super_admin / 测试 broker）
```

## 3. 生产部署（K8s）

### 3.1 Helm Chart 结构

```
deploy/helm/alfq/
├── Chart.yaml
├── values.yaml
├── values.prod.yaml
└── templates/
    ├── _helpers.tpl
    ├── admin-api.yaml         # Deployment + Service + HPA + NetworkPolicy + ServiceMonitor
    ├── md-gateway.yaml
    ├── factor-svc.yaml
    ├── strategy-svc.yaml
    ├── risk-svc.yaml
    ├── oms.yaml
    ├── web.yaml
    ├── jupyterhub.yaml         # 子 chart：jupyterhub/jupyterhub
    ├── ingress.yaml            # Caddy / nginx-ingress
    ├── secrets-vault.yaml      # Vault Agent Injector annotations
    └── network-policies.yaml
```

### 3.2 命名空间

- `alfq-system`：admin-api / web / 监控 / vault
- `alfq-trading`：md-gateway / strategy-svc / risk-svc / oms / factor-svc
- `alfq-research`：jupyterhub + notebook pods（per-tenant 进一步隔离用 namespace 或 NetworkPolicy）

### 3.3 关键配置

- HPA：admin-api / factor-svc / strategy-svc 按 CPU 70% + 自定义指标（订单速率）
- PDB：每服务 minAvailable=1
- Anti-affinity：行情/策略关键 pod 跨节点
- 资源 requests/limits 必须显式设置
- 镜像 ImagePullPolicy=Always（生产用 digest pin）

### 3.4 持久化

- PG：CloudSQL / RDS / 自建 patroni HA
- CH：3 副本 cluster（zookeeper or keeper）
- Redis：Sentinel 或 cluster
- NATS：3 replica + JetStream 持久化
- MinIO：分布式 4-8 节点 + erasure coding；或直接用 S3

### 3.5 Ingress / TLS

- Caddy / cert-manager 自动 Let's Encrypt
- HSTS、CSP 头由 Caddy 注入
- 仅暴露：`app.alfq.io`（前端）、`api.alfq.io`（admin-api）、`notebook.alfq.io`（JupyterHub）

## 4. CI/CD

### 4.1 GitHub Actions（示例 workflow）

```
.github/workflows/
├── ci.yml                  # PR：lint / test / build
├── proto.yml               # buf lint + breaking 检查 + generate
├── release.yml             # tag → 构建镜像 → 推 registry → cosign 签
└── deploy.yml              # 推 ArgoCD 仓库 / 触发同步
```

### 4.2 流水线步骤

```
1. checkout
2. setup go/python/node + cache
3. proto: buf lint + buf breaking
4. go: go vet, golangci-lint, govulncheck, go test -race, go build
5. py: ruff, mypy, pytest（含 DSL parity）
6. web: pnpm install, pnpm typecheck, pnpm build, pnpm test
7. docker build (每服务) + trivy scan + cosign sign
8. push images
9. update Helm values.yaml image tag → 推到 ArgoCD repo
```

## 5. 配置与秘钥

### 5.1 配置

- 业务配置 YAML 进入 ConfigMap
- 敏感字段不入 ConfigMap，走 Vault Agent

### 5.2 Vault Agent

Pod annotation：

```yaml
vault.hashicorp.com/agent-inject: "true"
vault.hashicorp.com/role: "alfq-svc-admin-api"
vault.hashicorp.com/agent-inject-secret-jwt: "kv/system/jwt/current"
```

容器启动前 `/vault/secrets/jwt` 自动写入。

## 6. 监控告警

### 6.1 Grafana 看板

- **总览**：所有服务可用性 / 错误率 / 延迟
- **交易**：tick rate / order rate / 成功率 / PnL
- **策略**：每策略实时持仓、信号数、PnL
- **风控**：拒单率 / 熔断状态 / 告警事件
- **基础**：CPU / 内存 / GC / goroutine

### 6.2 告警规则（Prometheus）

```yaml
groups:
- name: trading
  rules:
  - alert: BrokerDisconnected
    expr: md_connection_state == 0
    for: 30s
    labels: { severity: critical }
  - alert: OrderRejectRateHigh
    expr: rate(order_reject_total[5m]) / rate(order_submit_total[5m]) > 0.3
    for: 2m
  - alert: KillSwitchTriggered
    expr: risk_breaker_state{state="halted"} > 0
    for: 0s
    labels: { severity: page }
  - alert: ServiceDown
    expr: up{job=~"alfq-.*"} == 0
    for: 1m
```

### 6.3 通道

- P0：PagerDuty / 电话
- P1：Telegram + 钉钉
- P2：邮件

## 7. 日志

- 应用 stdout JSON → Promtail → Loki

## 10. 灰度发布

- 每次发布先到 paper 环境运行 N 小时无报错
- 生产分批：10% → 50% → 100%，每阶段观察 30 分钟
- 任何 P1 告警立即回滚（Helm rollback / ArgoCD rollback）

## 11. 容量规划

| 容量 | 计算依据 |
|---|---|
| md-gateway | 每经纪连接 ~50 MB；按品种数 + tick 速率 |
| CH | tick: 大约 80B/行 压缩后；100 symbols × 1tick/s × 86400s × 365 ≈ 数 TB/年/租户 |
| 策略 | 每部署 200 MB + 模型大小 |
| Redis | < 5 GB（缓存 + 会话 + 锁） |

## 12. 灾备

- 主备机房（伦敦/纽约/东京），就近经纪商
- 异地复制：CH cross-cluster replicate；PG 物理流复制
- 切换演练每半年一次

## 附录 B. 灾难恢复（DR）演练手册

### B.1 RPO / RTO 目标

| 系统 | RPO | RTO |
|---|---|---|
| PG 主数据 | 5 分钟 | 30 分钟 |
| ClickHouse | 1 小时 | 1 小时 |
| Redis（无状态缓存） | 不保证 | 5 分钟（重建） |
| NATS JetStream | 1 分钟 | 15 分钟 |
| 对象存储（MinIO/S3） | 0（多副本） | 即时 |
| 配置（Vault） | 即时（HA） | 5 分钟 |

### B.2 演练频率

| 演练 | 频率 |
|---|---|
| PG 备份恢复 | 每月 |
| CH 备份恢复 | 每季度 |
| 整体跨 AZ 切换 | 每半年 |
| 跨区域容灾 | 每年 |
| Vault unseal | 每季度 |
| KillSwitch 演练 | 每月 |

### B.3 标准演练剧本

#### B.3.1 PG 主库故障切换

```
1. 公告：演练开始时间 / 影响范围（status page 内部）
2. 模拟主库 down：systemctl stop postgres 或云端 reboot
3. 观察从库自动 promote（pg_auto_failover / patroni）
4. 验证：admin-api / oms 等服务自动重连
5. 检查：未完成事务 / outbox 滞留 / saga 状态
6. 恢复：旧主重建为新从
7. 复盘：耗时 / 数据丢失 / 客户感知
```

验收：RTO ≤ 30 min，无数据丢失（Outbox + WAL streaming 保证）。

#### B.3.2 ClickHouse 节点恢复

```
1. 模拟节点 down，清空数据目录
2. 从备份（clickhouse-backup）恢复至最近副本
3. 启动副本，等同步追平
4. 校验：行数 / 关键查询结果
5. 重新加入分布式表
```

#### B.3.3 跨区域容灾切换

```
1. DNS 切流到备区域
2. PG 备库 promote
3. CH 拉起备区域副本
4. NATS JetStream 跨区镜像追上
5. 各服务在备区域启动
6. 联调验证下单链路
7. 切回：反向操作
```

### B.4 备份验证

每次备份完成后自动：
- 抽样 1% 数据校验完整性
- 月度抽 1 个完整备份做"恢复演练"（在隔离环境）
- 报告存 `docs/backup-validation/`

### B.5 数据损坏应急

```
1. 检测：CRC 失败 / 业务校验失败 / 用户报错
2. 隔离：只读模式
3. 评估：损坏范围（事件 / 行 / 表 / 库）
4. 决策：从备份回滚 vs 数据修复脚本
5. 执行：附详细 SQL + 双人 review
6. 验证 + 解封
7. 复盘 + 防御机制
```

### B.6 安全事件 DR

机密泄露 / 入侵嫌疑：
1. 立即轮换所有秘钥（Vault revoke + reissue）
2. 失效所有 refresh_token
3. 强制所有用户重登 + 重设密码
4. 审计最近 30 天访问
5. 法务 / 监管通知（如适用）

### B.7 演练记录

每次演练写 `docs/dr-runs/YYYY-MM-DD-<topic>.md`：
- 演练目标 / 步骤 / 实测时间 / 偏离预期 / 改进项
