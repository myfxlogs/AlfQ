## 2. 开发环境（docker-compose）

`deploy/compose/docker-compose.yml` 至少包含：

```yaml
services:
  postgres:    { image: postgres:17, env: ..., volumes: pg-data }
  clickhouse:  { image: clickhouse/clickhouse-server:25, volumes: ch-data }
  redis:       { image: redis:8-alpine }
  nats:        { image: nats:2.10, command: -js }
  minio:       { image: minio/minio, command: server /data }
  vault:       { image: hashicorp/vault, command: server -dev }       # dev 用 -dev
  prometheus:  { image: prom/prometheus, volumes: cfg }
  grafana:     { image: grafana/grafana, volumes: data }
  loki:        { image: grafana/loki }
  tempo:       { image: grafana/tempo }
  jaeger:      { image: jaegertracing/all-in-one }
  jupyterhub:  { build: ./notebook }
  trading-core:   { build: ../../backend/go/cmd/trading-core }
  md-gateway:  { build: ../../backend/go/cmd/md-gateway }
  quant-engine:  { build: ../../backend/go/cmd/quant-engine }
  assistant-svc: { build: ../../backend/go/cmd/assistant-svc }
  web:         { build: ../../frontend }
```

启动：

```bash
make dev-up         # docker compose -f deploy/compose/docker-compose.yml up -d
make dev-migrate    # 跑 PG/CH 迁移
make dev-seed       # 种子数据（默认 super_admin / 测试 broker）
```

## 3. 生产部署（单机 docker-compose）

### 3.1 主机要求

| 项 | 规格 |
|---|---|
| OS | Ubuntu 24.04 LTS / Debian 12 |
| CPU | ≥ 16 核 |
| 内存 | ≥ 32 GB |
| 磁盘 | SSD ≥ 500 GB（RAID 1 推荐） |
| 网络 | 固定公网 IP，1 Gbps |
| 软件 | Docker Engine ≥ 26 + docker compose v2 |

### 3.2 编排文件

唯一生产 compose 文件：`deploy/docker-compose.prod.yml`，包含：

```
services:
  # 业务服务（5 个）
  trading-core   :8080  （build: backend/go/cmd/trading-core）
  md-gateway     :9001  （build: backend/go/cmd/md-gateway）
  quant-engine   :9002  （build: backend/go/cmd/quant-engine）
  assistant-svc  :9003  （build: backend/go/cmd/assistant-svc）
  frontend       :80    （build: frontend）

  # 基础设施
  postgres:17    :5432
  clickhouse:25  :8123 / :9000
  redis:8        :6379
  nats:2.10      :4222 / :8222
  vault          :8200
  prometheus     :9090
  grafana        :3000
  loki           :3100
  tempo          :3200
```

所有服务：
- `restart: unless-stopped`
- 显式 `healthcheck`
- `depends_on` + `condition: service_healthy`
- volume 命名持久化

### 3.3 部署命令

```bash
# 首次部署
git clone https://github.com/<org>/alfq.git /opt/alfq
cd /opt/alfq/deploy
cp .env.example .env  # 编辑密钥
docker compose -f docker-compose.prod.yml up -d --build

# 更新
cd /opt/alfq && git pull
cd deploy && docker compose -f docker-compose.prod.yml up -d --build

# 查看状态
docker compose -f docker-compose.prod.yml ps
docker compose -f docker-compose.prod.yml logs -f --tail=100 trading-core
```

### 3.4 持久化（Docker named volume）

| 服务 | volume | 备份策略 |
|---|---|---|
| postgres | `deploy_pg_data` | 每日 `pg_dump` → S3 |
| clickhouse | `deploy_ch_data` | 每日 `BACKUP TABLE` → S3 |
| redis | `deploy_redis_data` | AOF + RDB |
| nats | `deploy_nats_data` | JetStream 持久化目录 |
| vault | `deploy_vault_data` | 每日 raft snapshot |
| prometheus | `deploy_prom_data` | 不备份（指标短周期） |
| grafana | `deploy_grafana_data` | 每周备份 dashboards |

### 3.5 反向代理 / TLS

生产前置 **Caddy**（推荐）或 **nginx**：

- 80/443 暴露到公网
- 反代到内部 `frontend:80`（compose 内网）
- Caddy 自动签发 Let's Encrypt（`app.alfq.io`、`api.alfq.io` 共用一个证书或分开）
- 强制 HTTPS + HSTS + CSP 由 Caddy 注入

示例 `Caddyfile`：

```
app.alfq.io {
  reverse_proxy localhost:80
  header Strict-Transport-Security max-age=31536000
}
```

注：`trading-core:8080` 不直接暴露公网，前端通过 nginx 容器内 `/api/` 反代访问。

## 4. CI/CD

### 4.1 GitHub Actions

```
.github/workflows/
├── ci.yml                  # PR：lint / test / build
├── proto.yml               # buf lint + breaking 检查 + generate
├── release.yml             # tag → 构建镜像 → 推 registry + cosign sign
└── deploy.yml              # 通过 SSH 触发生产机 docker compose pull/up
```

### 4.2 流水线步骤

```
1. checkout
2. setup go/python/node + cache
3. proto: buf lint + buf breaking
4. go: go vet, golangci-lint, govulncheck, go test -race, go build
5. py: ruff, mypy, pytest（含 DSL parity）
6. web: pnpm install, pnpm typecheck, pnpm build, pnpm test
7. docker build (每服务) + trivy scan + cosign sign
8. push images 到 ghcr.io / private registry
9. deploy.yml（main / tag 触发）：
   - SSH 到生产机：appleboy/ssh-action
   - 执行：cd /opt/alfq && git pull && cd deploy && docker compose pull && docker compose up -d
```

生产机直接从 git 拉源码 + registry 拉镜像，不走 GitOps（ArgoCD 不适用单机）。

## 5. 配置与秘钥

### 5.1 配置

- 业务配置 YAML 进入 ConfigMap
- 敏感字段不入 ConfigMap，走 Vault Agent

### 5.2 Vault 集成（单机模式）

单机部署下 **Vault 与业务服务在同一 compose 网络**。无 K8s Agent Injector，改为：

**方式 A（推荐）：业务服务启动时主动拉取**
- 服务通过 `VAULT_ADDR=http://vault:8200` + `VAULT_TOKEN`（或 AppRole）启动
- 启动时 `vault read kv/system/jwt/current` 拉取 → 内存
- 周期性轮询续期

**方式 B：vault-agent sidecar（compose service）**
- 单独运行 `vault-agent` 容器，把秘钥渲染到共享 volume
- 业务服务 mount 该 volume 读取 `/etc/secrets/jwt`

**.env 文件管理**（开发/单机简化）：
- `deploy/.env` 存储 DB password、JWT seed 等
- 不进 git（`.gitignore` 已配置）
- 通过 `chmod 600` 限制权限
- 生产应迁移到 Vault；过渡期允许 .env

## 6. 监控告警

### 6.1 Grafana 看板

- **总览**：所有服务可用性 / 错误率 / 延迟
- **交易**：tick rate / order rate / 成功率 / PnL
- **策略**：每策略实时持仓、信号数、PnL
- **风控**：拒单率 / 熔断状态 / 告警事件
- **基础**：CPU / 内存 / GC / goroutine

### 6.2 告警规则（Prometheus）

```yaml
groups:
- name: trading
  rules:
  - alert: BrokerDisconnected
    expr: md_connection_state == 0
    for: 30s
    labels: { severity: critical }
  - alert: OrderRejectRateHigh
    expr: rate(order_reject_total[5m]) / rate(order_submit_total[5m]) > 0.3
    for: 2m
  - alert: KillSwitchTriggered
    expr: risk_breaker_state{state="halted"} > 0
    for: 0s
    labels: { severity: page }
  - alert: ServiceDown
    expr: up{job=~"alfq-.*"} == 0
    for: 1m
```

### 6.3 通道

- P0：PagerDuty / 电话
- P1：Telegram + 钉钉
- P2：邮件

## 10. 发布与回滚（单机）

- **发布**：先在 staging 运行 ≥ 24 小时无报错，再发布生产
- **生产部署**：`docker compose pull && docker compose up -d`（滚动重启每个服务，几秒中断）
- **回滚**：记住上一个 image tag，`IMAGE_TAG=<old> docker compose up -d <svc>`
- **任何 P1 告警立即回滚**，并在 staging 复现/调查

## 11. 容量规划（单机基线）

基于 16C/32G/500GB SSD 主机：

| 服务 | 内存预算 | 计算依据 |
|---|---|---|
| trading-core | 2 GB | API + OMS + 风控合并 |
| md-gateway | 2-4 GB | 每经纪连接 ~50 MB，多个品种 + tick 缓冲 |
| quant-engine | 4-8 GB | 因子 + 策略 + ONNX 模型 |
| assistant-svc | 1 GB | LLM 是云端 |
| postgres | 4 GB shared_buffers | |
| clickhouse | 8 GB | 列式压缩 + cache |
| redis | 2 GB | 缓存 + 会话 + 锁 |
| nats | 1 GB | JetStream |
| 监控栈 | 4 GB | prom + grafana + loki + tempo |
| **预留** | ~4 GB | 系统 + 突发 |

磁盘：
- ClickHouse 增长：100 symbols × 1tick/s × 86400s = ~7M 行/天/租户，压缩后 ~500MB/天
- 500GB 可支撑：约 100-200 个租户 × 12 个月（取决于 tick 密度）
- 满足 §0 当前规模目标（< 100 租户）

## 12. 灾备（单机方案）

单机不做实时主备切换，采用 **冷备 + 快速重建** 策略：

### 12.1 备份层次

| 层 | 内容 | 频率 | 目标 |
|---|---|---|---|
| L1 | PG/CH 逻辑导出 | 每日 | 异地对象存储（S3） |
| L2 | Docker volume tar | 每周 | 异地对象存储 |
| L3 | 整机 OS 快照 | 每月 | 云盘快照 |
| L4 | 代码 + 配置 | 持续 | Git remote |

### 12.2 故障场景与恢复

| 故障 | RTO | 操作 |
|---|---|---|
| 服务进程崩溃 | < 30s | `restart: unless-stopped` 自动拉起 |
| 单服务镜像错误 | < 5min | `docker compose up -d <svc>` 回滚镜像 tag |
| 主机重启 | < 5min | `docker compose up -d` 全栈拉起 |
| 主机硬件故障 | < 30min | 备用机 + 异地备份恢复 |
| 数据损坏 | < 1h | 从 L1 备份恢复 PG/CH |
| 安全事件 | < 1h | 全栈 down + 镜像重建 + 秘钥轮换 |

### 12.3 备用机准备

- 1 台同规格主机，定期从生产 sync `deploy/` 目录和最近备份
- DNS TTL ≤ 300s 便于切换
- 切换演练每季度一次

## 附录 B. 灾难恢复（DR）演练手册

### B.1 RPO / RTO 目标

| 系统 | RPO | RTO |
|---|---|---|
| PG 主数据 | 5 分钟 | 30 分钟 |
| ClickHouse | 1 小时 | 1 小时 |
| Redis（无状态缓存） | 不保证 | 5 分钟（重建） |
| NATS JetStream | 1 分钟 | 15 分钟 |
| 对象存储（MinIO/S3） | 0（多副本） | 即时 |
| 配置（Vault） | 即时（HA） | 5 分钟 |

### B.2 演练频率

| 演练 | 频率 |
|---|---|
| PG 备份恢复 | 每月 |
| CH 备份恢复 | 每季度 |
| 整体跨 AZ 切换 | 每半年 |
| 跨区域容灾 | 每年 |
| Vault unseal | 每季度 |
| KillSwitch 演练 | 每月 |

### B.3 标准演练剧本

#### B.3.1 PG 数据恢复（单机冷备场景）

```bash
1. 检查：docker compose logs postgres | tail -50，确认损坏范围
2. 停服：docker compose stop trading-core quant-engine assistant-svc
3. 从 S3 拉取最近 pg_dump：aws s3 cp s3://alfq-backup/pg/latest.sql.gz .
4. 恢复：gunzip < latest.sql.gz | docker compose exec -T postgres psql -U alfq
5. 启服：docker compose up -d trading-core quant-engine assistant-svc
6. 验证：账户登录 / 订单查询 / NATS outbox 重发
7. 复盘：耗时 / 丢失区间（最近备份后的数据需从 broker 重拉）
```

验收：RTO ≤ 1 小时；RPO = 最后一次成功备份时间点（默认 24 小时内）。

#### B.3.2 ClickHouse 数据恢复

```bash
1. docker compose stop clickhouse
2. docker volume rm deploy_ch_data（或备份后清空）
3. docker compose up -d clickhouse
4. 从 S3 拉取备份、导入：clickhouse-client --query "RESTORE ... FROM S3(...)"
5. 校验：SELECT count() FROM ticks_*
```

#### B.3.3 备用机切换（单机完整重建）

```bash
1. 备用机 SSH：git clone /opt/alfq（或 git pull）
2. 从 S3 拉取最近 PG/CH 备份 → docker volume 预热
3. cp /backup/.env /opt/alfq/deploy/.env
4. docker compose -f docker-compose.prod.yml up -d --build
5. DNS 切换到备用机 IP（TTL 300s）
6. 验证：前端可访问 / 下单链路 OK
7. 原主机修复后反向同步，下次演练反向
```

目标：RTO ≤ 30 分钟。

### B.4 备份验证

每次备份完成后自动：
- 抽样 1% 数据校验完整性
- 月度抽 1 个完整备份做"恢复演练"（在隔离环境）
- 报告存 `docs/backup-validation/`

### B.5 数据损坏应急

```
1. 检测：CRC 失败 / 业务校验失败 / 用户报错
2. 隔离：只读模式
3. 评估：损坏范围（事件 / 行 / 表 / 库）
4. 决策：从备份回滚 vs 数据修复脚本
5. 执行：附详细 SQL + 双人 review
6. 验证 + 解封
7. 复盘 + 防御机制
```

### B.6 安全事件 DR

机密泄露 / 入侵嫌疑：
1. 立即轮换所有秘钥（Vault revoke + reissue）
2. 失效所有 refresh_token
3. 强制所有用户重登 + 重设密码
4. 审计最近 30 天访问
5. 法务 / 监管通知（如适用）

### B.7 演练记录

每次演练写 `docs/dr-runs/YYYY-MM-DD-<topic>.md`：
- 演练目标 / 步骤 / 实测时间 / 偏离预期 / 改进项
