# 16 - 测试与质量保证

> 测试体系四层：**单测 / 契约 / 集成 / E2E + 性能 + 故障注入 + 一致性**。所有 AI Agent 产出代码必须以测试为先（TDD 优先），CI 强制覆盖率与基准。

## 1. 测试金字塔

```
                ┌─────────────┐
                │   E2E 5%    │  Playwright + 真实依赖
                └──┬──────────┘
              ┌────┴──────────┐
              │ Integration  20% │  testcontainers 拉真实 PG/CH/Redis/NATS
              └─────┬────────────┘
            ┌───────┴────────────┐
            │  Contract 5%       │  buf breaking + 跨语言一致性
            └─────────┬──────────┘
       ┌─────────────┴──────────────┐
       │    Unit 70%                 │  纯函数 / mock 依赖
       └─────────────────────────────┘
```

附加：**Performance、Chaos、DSL Parity、Backtest-vs-Live Parity** 平行运行。

## 2. 覆盖率门禁

| 范围 | 最低 | 核心模块 |
|---|---|---|
| Go 整体 | 60% | factor/dsl, oms/statemachine, risk/rules, signal/onnx ≥ 90% |
| Python 整体 | 70% | factor/dsl, backtest, metrics ≥ 90% |
| TypeScript 整体 | 50% | auth / order 流程 ≥ 80% |

**禁止用注释/排除项目**降低覆盖率分母。覆盖率不允许相对上一次下降。

## 3. 单元测试

### 3.1 Go

- 工具：`testing` + `testify` + `gomock`/`mockery`
- 模式：**Table-driven**

```go
func TestEMA(t *testing.T) {
    cases := []struct {
        name string
        in   []float64
        n    int
        want []float64
    }{
        {"basic_5", []float64{1,2,3,4,5,6}, 3, []float64{NaN, NaN, 2, 3, 4, 5}},
    }
    for _, tc := range cases {
        t.Run(tc.name, func(t *testing.T) { ... })
    }
}
```

- `t.Parallel()` 默认开启
- race detector 必须过：`go test -race -count=1 ./...`
- mock：基于 interface 自动生成 `go:generate mockery`

### 3.2 Python

- pytest + pytest-asyncio + pytest-cov
- fixtures 集中 `conftest.py`
- 参数化用 `@pytest.mark.parametrize`
- 不写 `print`，断言用 `assert ... == approx(...)` 处理浮点

### 3.3 TypeScript

- Vitest，组件用 `@testing-library/react`
- mock 用 `vi.mock` 自动 hoist
- 不写 snapshot（脆弱），写显式断言

## 4. 契约测试

### 4.1 Proto 契约

- `buf breaking --against main` 在 CI 必跑
- proto 变更走 ADR，引入新 `vN` 而非破坏旧

### 4.2 跨语言 schema 一致性

`schemas/` 目录存 JSON Schema / Proto-derived，Go/Py/TS 各自生成绑定后跑：

```
ContractTest:
  - 用 Go 序列化样本 → Py 反序列化 → 字段断言
  - 用 Py 序列化样本 → TS 反序列化 → 字段断言
```

样本存 `testdata/contracts/*.bin`。

### 4.3 DSL 跨语言 Parity

最关键。`research/tests/test_dsl_parity.py`：

- 加载 `testdata/golden/dsl_cases.yaml`：30+ 个表达式
- 对每个表达式：
  - Python 流式引擎跑结果
  - Python Polars 批量引擎跑结果
  - 通过 Connect RPC 调 Go 的 `FactorService.PreviewFactor` 跑结果
- 逐点比较，max abs diff < 1e-9
- CI 必跑

样例 yaml：

```yaml
- name: ema_basic
  expr: "ema($close, 20)"
  fixture: golden_bars_EURUSD_2024_01.parquet
  tolerance: 1e-9
- name: zscore_cross
  expr: "zscore($close, 100) > 2 ? 1 : 0"
  fixture: ...
```

### 4.4 回测 vs 实盘 Parity

`research/tests/test_backtest_live_parity.py`：

- 录制 paper 账户一周的 tick + 订单流
- 用相同数据跑事件驱动回测
- 比对：每日 PnL、净值曲线、成交数
- 偏差 > 1% 失败

CI 周期性跑（非每次 PR）。

## 6. E2E 测试（Playwright）

### 6.1 关键路径

| 用例 | 步骤 |
|---|---|
| 登录 + TOTP | 输入凭据 → TOTP → 进入 dashboard |
| 创建因子 | 填表达式 → 校验 → 预览 → 提交 |
| 部署策略 | 选策略 → 选账户 → 配参 → TOTP → 部署 |

### 6.2 数据准备

- E2E 前置：每用例独立租户 + 测试账户（fake broker）
- `beforeAll` 通过 admin API 创建数据，`afterAll` 清理

### 6.3 录像与失败回放

- Playwright trace + screenshot on failure
- CI 上传 artifact，PR 评论附链接

## 7. 性能基准

### 7.1 Go bench

```
go test -run='^$' -bench=. -benchmem -count=5 ./internal/factor/dsl/...
```

每个核心包必须有 benchmark：
- DSL 算子树评估：< 10µs/bar（10 因子）
- ONNX 推理：模型大小 1 MB 时 < 1ms
- 风控检查：< 1ms
- 订单状态机一次转移：< 50µs

CI 周期性跑 bench，结果存到 `bench-history/`，回归 > 20% 失败。

工具：[benchstat](https://pkg.go.dev/golang.org/x/perf/cmd/benchstat) 比较 PR 与 main。

### 7.2 负载测试

[k6](https://k6.io) 脚本位于 `tests/load/`：

- `login.js`：100 RPS 持续 5min
- `place_order.js`：50 RPS 持续 10min
- `stream_quotes.js`：1000 并发 SSE 持续 10min

每次 release 候选必跑，p99 / 错误率不可回退。

### 7.3 容量基准

记录在 `docs/benchmarks.md`（AI Agent 后续维护）：

- 单实例 admin-api：稳态 500 RPS p99 < 100ms
- md-gateway：1000 tick/s/connection 稳定
- factor-svc：100 因子 × 100 品种 × 1m bar = 10000 计算/分钟 < 50% CPU

## 8. 故障注入 / 混沌

### 8.1 工具

- Go 测试期：自研 `chaos` 中间件（注入延迟/错误/丢包）
- 集成期：[toxiproxy](https://github.com/Shopify/toxiproxy) 模拟网络问题
- K8s 期：[Chaos Mesh](https://chaos-mesh.org/) 或 [LitmusChaos]

### 8.2 必跑场景

| 场景 | 期望行为 |
|---|---|
| PG 短暂不可用 (10s) | 服务恢复后业务继续，pending 订单不丢 |
| CH 写入失败 | 缓冲队列落盘，恢复后回放 |
| NATS 消费 lag 10000+ | 慢消费者告警但不丢消息 |
| mtapi 断连 30s | 重连，pending 订单查询状态 |
| Redis 不可用 | 降级（缓存失效，幂等用 DB） |
| 一个实例 OOM | K8s 重启，无脏数据 |
| 时钟跳变 (+5min/-5min) | 不影响交易时段判断（用 NTP） |

每月跑一次完整套件，结果记录到 `docs/chaos-runs/`。

## 10. 测试数据管理

### 10.1 黄金数据集

- 位置：`backend/go/internal/testdata/golden/` + `research/tests/fixtures/`
- 格式：Parquet（数据）/ YAML（用例）/ JSONL（订单流）
- **版本化**：用 Git LFS 或 DVC，禁止直接进仓库
- **不可改**：修改需附原因 + 评审

### 10.2 合成数据生成

`research/alfq_research/synth/`：

- `synth_bars()`：可控波动率/趋势/跳空生成 bar
- `synth_ticks()`：从 bar 反推合理 tick
- `synth_orders()`：生成订单流（含部分成交、撤单、拒单）

固定 seed 保证可重现。

### 10.3 数据脱敏

测试中绝不用生产数据；如必须从生产取样本，必须：
1. 去除 user_id / email / 账户号
2. 数值加 ±5% 噪声
3. 通过法务审批

## 11. 持续集成（CI）流水线

```
PR → CI 触发
  1. checkout + setup matrix (Go/Py/Node)
  2. lint (golangci-lint / ruff / eslint)
  3. proto: buf lint + buf breaking
  4. unit:  Go test -race / pytest / vitest
  5. contract: cross-lang parity
  6. integration: testcontainers
  7. security: gosec/bandit/trivy/gitleaks
  8. coverage report → PR comment
  9. perf delta (benchstat) → PR comment
 10. build images
 11. E2E (Playwright) on ephemeral env
```

任一步失败 → 阻止合并。

## 12. 测试编写规范

- 测试名：`Test<UnitUnderTest>_<Condition>_<Expectation>`
- 一个测试一个断言点（多点用 sub-test）
- **Arrange / Act / Assert** 三段式
- 不依赖测试间顺序
- 时间相关用 fake clock（Go: `clockwork`；Py: `freezegun`；TS: `vi.useFakeTimers`）
- 网络相关：mock 或 testcontainers，**绝不打外网**

## 13. AI Agent 写测试要求

- **TDD 优先**：先写测试，运行红，再实现，运行绿，再重构
- 每个 PR 必须包含：
  - 至少 1 个失败重现测试（修 bug 时）
  - 至少覆盖所有公开接口的快乐路径 + 1 个错误路径
  - 黄金数据有变更时附 diff 解释
- 不允许 `t.Skip()` / `pytest.skip` 进 main，必须 issue tracker 跟踪
- 禁止删除已有测试（除非显式弃用 + 评审）
