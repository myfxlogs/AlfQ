## 3. 关键发现

### 3.1 🔴 高风险：Tempo 运行 RC 预发布版

`grafana/tempo:latest` 当前解析到 **v3.0.0-rc.1**，这是一个**预发布候选版**，不是稳定版。

**风险：**
- RC 版可能包含未发现的 bug，不适合生产运行
- `latest` tag 在 milestone 逾期期间会指向预发布版本
- 当前稳定版是 **v2.10.5**

**修复方案：**
```yaml
# deploy/docker-compose.prod.yml
tempo:
  image: grafana/tempo:2.10.5  # 锁定到稳定版
```

### 3.2 🟡 中风险：`latest` tag 不可复现

Prometheus、Grafana、Loki、Tempo 均使用 `latest` tag，导致：
- 不同时间构建会拉取不同版本
- 无法做可复现的部署
- 意外升级到不兼容版本

**修复方案：** 全部锁定到当前实际版本号

### 3.3 🟡 中风险：ClickHouse 落后大版本

当前 `25.3-alpine`，最新 `26.1.12.23-stable`。ClickHouse 大版本升级通常需要：
- 确认 Go 驱动兼容性 (`clickhouse-go/v2` 是否支持 26.x)
- 检查 schema 兼容性

## 4. 升级执行顺序（推荐）

| 优先级 | 操作 | 预计工作量 | 备注 |
|---|---|---|---|
| P0 | **Tempo: latest → 2.10.5** | 5 分钟 | 立即执行，消除 RC 风险 |
| P0 | **所有 `latest` tag 锁定** | 15 分钟 | Prometheus/Grafana/Loki/Tempo |
| P1 | **PostgreSQL: 17 → 18** | 0.5 人天 | 见 docs/27 §9.6 备忘 |
| P1 | **Vault: 1.18 → 2.0.0** ⚠️ | 1 人天 | 主版本跳变，需验证 API 兼容 + dev mode |
| P2 | **ClickHouse: 25.3 → 26.1** | 0.5 人天 | 需验证 Go 驱动 + schema |
| P2 | **Alpine: 3.21 → 3.23** | 0.25 人天 | 改所有 Dockerfile + compose |
| P2 | **Nginx: 1.27 → 1.30.1** | 15 分钟 | 改 frontend/Dockerfile |
| P3 | **Node.js: 22 → 24（LTS）** | 0.5 人天 | 需验证前端构建兼容性；26 为非 LTS，推荐 24 |
| P3 | **PG 18 大版本统计保留** | 0.25 人天 | 升级后享受 pg_upgrade 统计保留特性 |
