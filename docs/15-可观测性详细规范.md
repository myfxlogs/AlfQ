# 15 - 可观测性详细规范

> 三大支柱：**Metrics（Prometheus）+ Logs（Loki）+ Traces（OTel/Tempo）**。本文给出 **统一的命名、字段、采样、告警** schema，所有服务必须遵守，CI 跑 schema lint。

## 1. 命名规则总览

| 类型 | 规则 |
|---|---|
| 指标名 | `alfq_<domain>_<noun>_<unit>` snake_case；计数器以 `_total` 结尾；时间以 `_seconds` 结尾 |
| 标签 | snake_case；基数受控（见 §3） |
| 日志字段 | 固定字段 + 业务字段；下划线命名 |
| trace span 名 | `<service>.<operation>` |

## 2. 指标 schema

### 2.1 强制公共标签（所有指标）

| 标签 | 必填 | 示例 |
|---|---|---|
| `service` | ✓ | `trading-core` |
| `env` | ✓ | `prod` / `paper` / `dev` |
| `instance` | ✓（自动） | `trading-core-7f8b-abc` |
| `version` | ✓ | `1.2.3` |

### 2.2 业务标签（按需）

| 标签 | 何时用 | 基数控制 |
|---|---|---|
| `tenant_id` | 业务指标 | 用租户 hash 桶或聚合 |
| `account_id` | 账户级 | 高基数，仅落 CH，不落 Prom |
| `broker` | 经纪商 | 低基数 ✓ |
| `symbol` | 行情/订单 | 中基数，用白名单 |
| `strategy_id` | 策略 | 中基数 |
| `code` | 错误码枚举 | 低基数 ✓ |
| `result` | success/fail | 低基数 ✓ |
| `method` | RPC 全名 | 中基数 ✓ |

**禁止**用 `client_order_id` / `user_email` / `ip` 当 Prom 标签。

### 2.4 直方图桶

统一桶（除非特殊说明）：

```

```

下单 e2e 用更精细：`[.01, .025, .05, .075, .1, .15, .2, .3, .5, 1, 2, 5]`。

## 3. 标签基数控制

- Prom 单指标活跃序列 ≤ 10 万，否则告警
- `symbol` 标签使用白名单（活跃品种列表），未在白名单的归 `other`
- `tenant_id` 高基数场景用 `tenant_bucket = hash(tid) % 16` 替代
- 全量按 `tenant_id` 拆分的统计落 ClickHouse，不落 Prom

## 4. 日志规范

> **格式选择**：日志输出统一使用 **结构化 JSON**（zap JSON encoder），**不使用 protobuf binary**。
> 原因：Loki/Promtail 链路只解析 JSON / logfmt；运维工具链（`grep` / `jq` / Grafana Logs UI）依赖文本可读；序列化 CPU 不是瓶颈。
> 这是一个有意识的边界划分：跨进程契约（RPC / NATS event）走 protobuf，进程内可观测性输出走 JSON。详见 doc 01 §2.1.1。

### 4.1 结构化字段（所有服务统一）

```json
{
  "ts": "2026-05-18T01:23:45.678Z",
  "level": "info | debug | warn | error | fatal",
  "service": "oms",
  "version": "1.2.3",
  "env": "prod",
  "instance": "oms-abc-1",
  "trace_id": "0af7651916cd43dd8448eb211c80319c",
  "span_id": "b7ad6b7169203331",
  "tenant_id": "...",
  "user_id": "...",
  "request_id": "...",
  "msg": "短信息（祈使句/陈述句）",
  "code": "ERR_CODE_OR_EVENT",          // 业务码
  "fields": { ... }                      // 业务自定字段
}
```

### 4.2 等级使用

| 级别 | 场景 |
|---|---|
| debug | 单 tick / 调试细节，仅 dev/paper 开启 |
| info | 业务事件：登录、下单、风控通过等正常路径 |
| warn | 可恢复异常、重试、对账差异 < 阈值 |
| error | 影响单请求失败但不影响系统 |
| fatal | 服务即将退出 |

不可逆破坏性错误必须用 error + 附带可定位的 fields。

### 4.3 脱敏

- 禁止打印：password / token / refresh_token / TOTP secret / Vault path 的全部值
- JWT 仅记录 jti 前 8 位
- 邮箱：`xxx@example.com` → `x***@example.com`
- IP：保留前 3 段，末段掩码

### 4.4 PII 字段标记

某些日志字段属于 PII，必须打 `pii=true` 标签便于 Loki 隔离归档：

```json
{"msg":"login","pii":true,"user_email_masked":"x***@example.com"}
```

### 4.5 错误日志必须包含

- `code`（业务错误码）
- `cause`（被包装的底层错误）
- `stack`（关键路径，仅 error/fatal）
- 必要时附 `payload_digest`（哈希，不要全量明文）

## 5. 分布式追踪（OpenTelemetry）

### 5.1 span 命名

- Connect/gRPC server：`<service>.<RPC全名>`（如 `trading-core.OrderService.PlaceOrder`）
- 客户端调用：`<target_service>.<op>`
- DB：`pg.query` / `ch.insert` / `redis.op` / `nats.publish`
- 外部：`mtapi.<method>`

### 5.2 必带属性

- `service.name` / `service.version` / `deployment.environment`
- `tenant.id` / `user.id`（对应租户/用户）
- `rpc.system=connect` / `rpc.method=...`
- `db.system / db.statement.hash`（不能放原文，放 sha256 前 12）
- `broker.code` / `symbol` / `order.id`（OMS 上下文）
- `error=true` + `error.code` + `error.message`（异常 span）

### 5.3 采样

- 默认 **Parent-based + ratio**：`prod=10%`、`paper=50%`、`dev=100%`
- **强制采样**：错误 span、敏感操作（KillSwitch / DeployStrategy / Login）
- 端到端订单链路：从 `trading-core.PlaceOrder` 到 `oms` 到 `trading-core` 到 `mtapi.Submit`，通过 `trace_id` 串联

### 5.4 上下文传播

- HTTP/Connect：`traceparent` header（W3C）
- NATS：消息 header 注入 `traceparent`
- 数据库：通过 application_name 附 trace_id 前 16 位

## 6. SLO 与告警

### 6.1 SLO 目标

| 服务 | 指标 | 目标（30d 滚动） |
|---|---|---|
| trading-core | 可用性（2xx/200 over total）| 99.9% |
| trading-core | p99 延迟 | < 200ms |
| oms | 下单成功率（非业务拒单）| 99.95% |
| oms | e2e p99 | < 500ms |
| md-gateway | 连接可用性 | 99.9% |
| md-gateway | tick → 发布 p99 | < 50ms |
| quant-engine | bar → 因子值 p99 | < 100ms |
| trading-core | check p99 | < 10ms |

### 6.2 错误预算

- 用 `(1 - SLO) × 30天` 计算预算
- 预算消耗 50% / 75% / 100% 分别告警
- 预算耗尽 → 冻结非紧急发布

### 6.3 告警规则表（Prometheus）

| 告警 | 表达式 | severity | 持续 |
|---|---|---|---|
| ServiceDown | `up{job=~"alfq-.*"} == 0` | critical | 1m |
| HighErrorRate | `rate(alfq_rpc_requests_total{code=~"5.*"}[5m]) / rate(alfq_rpc_requests_total[5m]) > 0.02` | warning | 5m |
| BrokerDisconnected | `alfq_md_connection_state == 0` | critical | 30s |
| TickLatencyHigh | `histogram_quantile(0.99, sum by(le)(rate(alfq_md_tick_latency_seconds_bucket[5m]))) > 0.2` | warning | 3m |
| OrderRejectSpike | `rate(alfq_order_submit_total{result="rejected"}[5m]) / rate(alfq_order_submit_total[5m]) > 0.3` | critical | 2m |
| OMSReconcileDiff | `alfq_oms_reconcile_diff_count > 0` | warning | 5m |
| OMSReconcilePersistent | `alfq_oms_reconcile_diff_count > 0` | critical | 15m |
| RiskBreakerActive | `alfq_risk_breaker_state > 0` | critical | 0s |
| KillSwitchOn | `alfq_risk_killswitch_active == 1` | page | 0s |
| LoginFailureSpike | `rate(alfq_auth_login_total{result="fail"}[5m]) > 10` | warning | 2m |
| PGSlowQuery | `histogram_quantile(0.99, ...) > 1` | warning | 5m |
| CHIngestLag | `time() - max(alfq_md_last_ingest_ts) > 60` | warning | 2m |
| VaultSealed | `vault_core_unsealed == 0` | critical | 30s |
| CertExpiringSoon | `(probe_ssl_earliest_cert_expiry - time()) / 86400 < 14` | warning | 1h |
| HighGoroutines | `go_goroutines > 5000` | warning | 10m |
| MemoryHigh | `container_memory_working_set_bytes / container_spec_memory_limit_bytes > 0.85` | warning | 10m |

完整规则文件存 `deploy/prometheus/rules/`，CI 跑 `amtool check-rules`。

### 6.4 告警分级与渠道

| Severity | 渠道 | 响应 SLA |
|---|---|---|
| `page` (P0) | PagerDuty 电话 + SMS | 5 分钟 |
| `critical` (P1) | PagerDuty + Telegram + 钉钉 | 15 分钟 |
| `warning` (P2) | Telegram + 钉钉 | 1 小时 |
| `info` (P3) | 邮件 / 看板 | 工作时间 |

### 6.5 静音/抑制规则

- 服务整体 Down → 抑制下游具体告警
- KillSwitch 主动触发期间 → 抑制订单拒单告警
- 计划维护窗口 → 提前在 Alertmanager 设静音

## 8. 日志聚合查询常用模板

LogQL 示例：

```
# 某次 trace 全链路日志
{service=~"alfq-.*"} |= "<trace_id>"

# 某租户所有错误
{service=~"alfq-.*"} | json | tenant_id="<tid>" | level="error"

# 某账户最近的下单
{service="oms"} | json | account_id="<aid>" | code="ORDER_SUBMIT"

# 跨租户访问尝试（安全审计）
{service="trading-core"} | json | code="TENANT_MISMATCH"
```
