## 9. PostgreSQL 18 升级评估

### 9.1 现状

- 当前版本：**PostgreSQL 17-alpine**（`postgres:17-alpine`）
- PG 17 发布：2024-09，预计 EOL：2029-11
- **PG 18 发布：2025-09-25**，当前最新 minor：**18.4**（2026-05-14），预计 EOL：2030-11
- 按 `docs/26 §1` 原则：**应使用最新稳定版**，当前 PG 17 属于需评估升级的版本

### 9.2 关键新特性（与 ALFQ 相关）

| 特性 | 对 ALFQ 的价值 |
|---|---|
| **异步 I/O (AIO)** | 高。量化系统高频 tick/订单写入，AIO（io_uring）在顺序扫描和 vacuum 场景性能提升最高 **3 倍**。 |
| **pg_upgrade 统计信息保留** | 中。未来 18→19 升级时，查询计划器统计信息不丢失，避免升级后性能断崖。 |
| **B-tree skip scan** | 中。多租户场景下 `(tenant_id, account_id, ...)` 复合索引常见，skip scan 可优化省略前缀等值的查询。 |
| **OR 条件索引优化** | 中。风控查询、订单筛选常带 OR，`WHERE a=X OR b=Y` 现在可走索引。 |
| **uuidv7()** | 低。若未来用 UUID 主键（当前用自增 INT），uuidv7 时间排序特性利于缓存。 |
| **虚拟生成列 (virtual generated columns)** | 低。可能用于派生字段，但当前 schema 未使用。 |
| **OAuth 认证** | 低。当前认证走 JWT + ConnectRPC，OAuth 是未来的可选扩展。 |
| **MD5 废弃 → SCRAM** | 低。当前密码存储已在应用层处理，PG 层认证用标准连接字符串。 |

### 9.3 升级收益

| 收益项 | 说明 |
|---|---|
| **性能提升** | AIO 对高并发 OLTP（订单、风控事件写入）和批量读取（因子计算）有直接收益 |
| **更长的支持周期** | +1 年 EOL（2030-11 vs 2029-11），减少未来紧急升级压力 |
| **更平滑的未来升级** | pg_upgrade 改进（--jobs 并行、--swap 目录交换、统计保留）降低 18→19 成本 |
| **查询优化器改进** | skip scan + OR 索引优化可减少手动索引维护工作 |

### 9.4 升级风险

| 风险项 | 评估 | 缓解措施 |
|---|---|---|
| **pgx/v5 兼容性** | 🟢 **低**。`jackc/pgx/v5` 对 PG 18 支持良好，Go 驱动层无已知 breaking change。 | 在 staging 环境跑 `go test` + 集成测试验证 |
| **sqlc 生成代码** | 🟢 **低**。sqlc 生成的是标准 SQL，PG 18 语法向后兼容，无已知问题。 | 重新生成并跑单元测试 |
| **RLS (Row-Level Security)** | 🟢 **低**。PG 18 未对 RLS 做 breaking change，当前多租户策略不受影响。 | 跑 RLS 相关集成测试 |
| **数据迁移时间** | 🟡 **中**。`pg_upgrade` 从 17→18 需停机。当前数据量：**空/测试数据**（M6.5+ 阶段），迁移成本极低。 | 选择维护窗口执行，或直接用 `pg_dump`/`pg_restore`（数据少时更快） |
| **Docker 镜像可用性** | 🟢 **低**。`postgres:18-alpine` 已发布。 | 已在 Docker Hub 验证可用 |
| **ClickHouse 集成** | 🟢 **无**。CH 通过 Go 驱动/HTTP 直接连接，不经过 PG，PG 版本不影响。 | — |
| **监控栈兼容性** | 🟢 **无**。Prometheus/Grafana 不直接读写 PG schema。 | — |
| **升级后 ANALYZE** | 🟢 **低**。PG 18 保留统计信息，升级后无需等待 ANALYZE 重建。 | 直接享受特性 |

### 9.5 建议

| 维度 | 结论 |
|---|---|
| **是否升级** | **建议升级**。PG 18 是稳定版，核心收益（AIO 性能、更长支持期）对量化系统有价值，风险可控。 |
| **升级时机** | 当前（M6.5+）或 **M7 开始前**。当前数据为空/测试数据，`pg_dump`/`pg_restore` 最快；若等到生产有实盘数据后再升级，成本会显著增加。 |
| **升级方式** | **方案 A（推荐）**：`docker compose down -v`，修改 compose 镜像为 `postgres:18-alpine`，重新 `up`。数据从种子脚本/备份重建。<br>**方案 B**：`pg_upgrade`（保留现有数据，适合未来数据量大时）。 |
| **工作量** | 约 **0.5 人天**：改镜像 tag → 重建 → 跑全套测试 → 更新 `docs/26` 和本文档。 |

### 9.6 升级执行备忘（Agent 施工用）

```bash
# 1. 备份当前 compose 配置（已做）
cd /opt/alfq/deploy

# 2. 修改镜像 tag（两处）
sed -i 's|postgres:17-alpine|postgres:18-alpine|g' docker-compose.prod.yml
cd .. && make go-test   # 确认 Go 测试通过

# 3. 重建（当前数据为空，直接 down -v 重建最快）
cd deploy
docker compose -f docker-compose.prod.yml down -v
docker compose -f docker-compose.prod.yml up -d --wait

# 4. 验证
curl -s http://localhost:8080/healthz | grep -q 200 && echo "OK"

# 5. 更新文档
cd /opt/alfq
# 更新 docs/26 §2.1：Go 表中的 PG 17 → 18
# 更新 docs/27 §5.1：postgres:17-alpine → 18-alpine
# 删除 docs/26 §4 中 PG 相关豁免（如有）
```
