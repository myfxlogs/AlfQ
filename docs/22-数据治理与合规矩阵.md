# 22 - 数据治理与合规矩阵

> 集中规定**数据分类、保留期、加密、访问控制、删除、GDPR/CCPA 数据主体权利、ML 模型治理**。AI Agent 实施新表/新字段时必须对照本文分类。

## 1. 数据分类（4 级）

| 级别 | 名称 | 例 | 处理要求 |
|---|---|---|---|
| **L4** | 极敏感 | 密码哈希、TOTP 密钥、Vault 凭据、私钥 | 仅加密存储（Vault）；日志/审计不打印；最小角色访问 |
| **L3+** | 敏感（协议限制） | broker 账号密码（MT 协议要求明文递交上游） | PG 磁盘加密 + RLS + 最小权限 + 审计；日志脱敏 |
| **L3** | 敏感（PII / 资金） | 邮箱、IP、手机、姓名、订单/资金/持仓、API Key | 静态加密；脱敏日志；RBAC 限制；可审计访问 |
| **L2** | 内部 | 策略 spec、因子表达式、tenant 配置 | 普通保护；租户隔离 |
| **L1** | 公开 | 品种属性、文档 | 无特殊限制 |

每张表/字段必须在 `docs/data-dictionary.md`（AI Agent 实施时维护）标注级别。

## 2. PII 字段矩阵

| 字段 | 级别 | 加密 | 日志策略 | 导出策略 |
|---|---|---|---|---|
| `users.email` | L3 | 否（hash 形式可索引） | 掩码 `x***@dom.com` | 主体导出包含；其他人脱敏 |
| `users.password_hash` | L4 | argon2id | 永不打印 | 永不导出 |
| `users.totp_secret` | L4 | Vault Transit | 永不打印 | 永不导出 |
| `refresh_tokens.token_hash` | L4 | sha256 存储 | 永不打印 | 永不导出 |
| `audit_logs.ip` | L3 | 否 | 末段掩码 | 主体可申请 |
| `audit_logs.user_agent` | L3 | 否 | 哈希 | 主体可申请 |
| `accounts.password` | L3+ | 明文（MT 协议要求明文递交上游；依赖 PG 磁盘加密 + RLS） | 脱敏（打印前 2 位 + `****`） | 永不导出 |
| `ai_messages.content` | L3 | 否 | redact PII | 主体可申请 |
| `orders.*` | L3 | 否 | 脱敏 client_id 前 8 位 | 主体可申请 |

## 3. 数据保留期矩阵

| 数据 | 保留期 | 归档 | 删除策略 |
|---|---|---|---|
| 用户账号 | 注销后 30 天软删 → 物理删除 | — | 强制 |
| 审计日志 | 5 年 | 90 天后 → S3 Object Lock | 不可删除 |
| 交易订单 | 7 年（合规） | 90 天后 → S3 | 7 年后清除 |
| 成交流水 | 7 年 | 同上 | 同上 |
| tick / bar 数据 | 2 年（按 plan 调） | 6 个月后 → S3 冷存 | 过期自动 drop partition |
| 因子值 | 2 年 | — | 同上 |
| 信号 | 2 年 | — | 同上 |
| 风控事件 | 5 年 | 1 年后 → S3 | 不可删 |
| 备份 | 30 天 | — | rotate |
| 会话 / refresh token | 7 天 / TTL | — | 自动过期 |
| Redis 缓存 | TTL | — | 自动 |
| ai_messages | 1 年 | 30 天后归档 | 主体请求可早删 |
| Notebook 用户数据 | 用户控制 | — | 用户删除 |

实施：
- ClickHouse `TTL toDateTime(ts) + INTERVAL ...`
- PG cron job 删除过期数据
- S3 Lifecycle policy 自动迁移到 Glacier

## 4. 加密策略

| 范围 | 策略 |
|---|---|
| 传输 | TLS 1.3，内部 mTLS |
| 静态 - 数据库 | 磁盘加密（云默认 / LUKS） |
| 静态 - 字段级 | L4 字段用 Vault Transit；L3 选用 |
| 静态 - 对象存储 | SSE-KMS（云）/ SSE-S3 |
| 备份 | GPG / KMS 加密 |
| 日志 | Loki 默认明文（脱敏后），归档加密 |

密钥轮换：
- TLS 证书：30 天
- JWT 签名 key：90 天
- 备份加密 key：180 天

## 6. GDPR / CCPA 数据主体权利

提供以下 RPC（trading-core 中）：

| 权利 | RPC | 说明 |
|---|---|---|
| 访问（Right to Access） | `UserService.ExportMyData` | 用户可下载自己的全部数据（JSON+ZIP） |
| 更正（Rectification） | `UserService.UpdateProfile` | 已存在 |
| 删除（Right to be Forgotten） | `UserService.RequestErasure` | 触发 30 天等待 + 软删 → 物理删 |
| 限制处理 | `UserService.RestrictProcessing` | 暂停账户但保留数据 |
| 数据可携（Portability） | 同 ExportMyData，标准 JSON | |
| 反对处理 | `UserService.OptOut` | 关闭 AI 助手等可选功能 |
| 撤回同意 | `UserService.RevokeConsent` | 同上 |

实施：
- 用户请求 → 工单 → 工程或 ops 执行
- 法定时限（GDPR 30 天 / CCPA 45 天）内完成
- 操作全程审计
- 删除范围：用户 PII + 自有数据；保留：法定保留的审计/交易记录（匿名化处理）

## 7. 同意管理

用户注册 / 加入新功能时记录 `user_consents`：

```sql
CREATE TABLE user_consents (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL,
  purpose     TEXT NOT NULL,        -- 'service_use' / 'ai_assistant' / 'analytics' / ...
  granted     BOOLEAN NOT NULL,
  version     TEXT NOT NULL,        -- 隐私政策版本
  ip          INET,
  granted_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

变更同意 → 影响功能可用性。

## 8. ML 模型治理

### 8.1 Model Card

每个 ONNX 模型必须有 Model Card（`s3://.../models/<id>/card.yaml`）：

```yaml
model_id: momentum_v1
strategy_id: ...
created_by: user@tenant
created_at: 2026-05-18
training:
  data_range: 2022-01-01..2024-12-31
  symbols: [EURUSD, GBPUSD]
  rows: 1234567
  features: [mom_20_60, atr_14, ...]
  algorithm: LightGBM
  hyperparams: {...}
metrics:
  train: {auc: 0.71, ...}
  validation: {auc: 0.65, ...}
limitations:
  - 不适用于高波动事件（NFP / FOMC）
  - 仅训练于流动性大币种
intended_use: 中频动量信号
hash_sha256: <onnx 文件哈希>
signed_by: <vault transit key id>
```

### 8.2 模型签名

- 上传 ONNX 时 Vault Transit 签名 hash
- 加载时验签
- 验签失败 → 拒绝部署 + 告警

### 8.3 漂移监控

部署后周期性：

- **数据漂移**：input feature 分布偏移（PSI / KS test）→ 触发再训练通知
- **预测漂移**：output 分布偏移
- **绩效退化**：滚动 Sharpe / Hit Rate 下降阈值
- 触发 → 通知 quant + 自动停 paper / 提示 live 评估

存表 `model_drift_metrics`，集成到 15 章监控。

### 8.4 模型生命周期

```
draft → validated → staged → live → deprecated → archived
```

- 升级版本：新 model_id，灰度替换（17 章 Feature Flag）
- 历史版本永久保留（合规追溯）

## 9. 数据质量

### 9.1 自动检查

- **完整性**：tick gap 检测（连续 N 分钟无 tick → 告警）
- **及时性**：CH `now() - max(ts) > 60s` → 告警
- **正确性**：bid > ask、负价格、跳变 > 10% → 标记异常
- **一致性**：CH 与 PG 关键统计每日对比

工具：`backend/go/internal/dataquality/` + Grafana 看板。

### 9.2 数据血缘

记录到 `data_lineage` 表：

```
表/视图 → 上游表 → 由哪个 ETL/物化视图 产出
```

便于变更影响分析。

## 10. 数据导出/导入

### 10.1 导出

- 租户管理员可导出本租户数据（GDPR 可携权 + 业务备份）
- 用户可导出自己数据（GDPR 访问权）
- 实施：trading-core `ExportData` 异步任务 → S3 临时 URL（24h）+ 邮件通知

### 10.2 导入

- 仅平台运维可导入（一次性数据迁移）
- 必须通过 schema 校验 + 配额检查
- 审计完整记录

## 11. 跨境数据

- 按 tenant 配置数据所在地（EU / US / APAC）
- 不同区域独立部署 + 数据复制策略
- 跨区传输需用户同意 + 加密
