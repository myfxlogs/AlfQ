# 05 - 多租户与权限设计

## 1. 多租户模型

### 1.1 隔离层次

| 层 | 隔离方式 | 备注 |
|---|---|---|
| 域名 | 共享域 + 路径 / 子域 | `app.alfq.io/t/{slug}` 或 `{slug}.alfq.io` |
| 应用 | JWT 携带 `tenant_id`，gRPC interceptor 注入 ctx | |
| PostgreSQL | 行级 RLS（02 章详述） | `current_setting('app.tenant_id')` |
| ClickHouse | Row Policy + 应用层强制 `WHERE tenant_id=?` | 每租户独立 CH user |
| Redis | Key 前缀 `t:{tenant_id}:...` | 同库逻辑隔离；高 SLA 租户可独立 db |
| 对象存储 | 桶前缀 `tenants/{tenant_id}/` + IAM 策略 | |
| Notebook | 每用户独立沙箱容器 + 卷 | 物理隔离 |
| Broker 账号 | **物理隔离**：不同租户绝不共用账号凭据 | |
| 监控 | Prometheus label `tenant_id`，Grafana 看板按租户过滤 | |

### 1.2 租户生命周期

```
[创建] → active → [暂停] suspended → [恢复] active
                                  ↓
                              [软删除] deleted
                                  ↓
                            [硬删除归档] purged（保留审计 5 年）
```

- 暂停：禁止登录、停所有部署、断开 broker 连接
- 软删除：30 天保留窗口可恢复
- 硬删除：清除 PG/CH 数据（保留审计冷归档），清除 broker 密码（accounts.password）

### 1.3 资源配额

每租户在 `tenants.plan` 决定：

| 字段 | 含义 |
|---|---|
| `max_users` | 最多用户数 |
| `max_strategies` | 策略数 |
| `max_accounts` | 经纪账户数 |
| `max_deployments` | 并发部署数 |
| `max_factors` | 因子数 |
| `max_backtest_per_day` | 每日回测次数 |
| `notebook_cpu` / `notebook_mem` | 沙箱配额 |
| `data_retention_days` | tick 保留天数 |

`admin-api` 在创建资源前检查；`risk-svc` 防止运行时超额。

## 2. RBAC 权限模型

### 2.1 实体

- **User**（用户）
- **Role**（角色，租户级或系统级）
- **Permission**（权限码，全局枚举）
- **ACL**（资源级覆盖：用户对单一资源的额外/限制权限）

### 2.2 权限码命名规范

`<resource>.<action>`：

```
tenant.manage
user.read       user.write       user.invite
role.manage
account.read    account.write    account.connect    account.disconnect
broker.read     broker.write
symbol.read
factor.read     factor.write     factor.approve
strategy.read   strategy.write   strategy.deploy    strategy.stop    strategy.kill
order.read      order.place      order.cancel
position.read
risk.read       risk.write       risk.kill_switch
backtest.run    backtest.read
audit.read
notebook.use
```

### 2.3 内置角色

| 角色 | 范围 | 默认权限 |
|---|---|---|
| `super_admin` | 系统 | 所有 |
| `tenant_admin` | 租户 | 除系统级外所有 |
| `trader` | 租户 | account.*, order.*, position.*, strategy.deploy/stop, risk.read |
| `quant` | 租户 | factor.*, strategy.write, backtest.*, notebook.use（不含 deploy） |
| `viewer` | 租户 | *.read |
| `risk_officer` | 租户 | risk.*, audit.read, strategy.kill |

### 2.4 ACL 覆盖

用户可被授予对**特定资源**的额外权限或拒绝：

```
acls(user_id, resource_type, resource_id, action, effect)
effect ∈ {allow, deny}
```

判定顺序：**deny ACL > allow ACL > Role 默认**。

### 2.5 后端鉴权流程

```
请求 → AuthInterceptor 解 JWT → 注入 ctx{user_id, tenant_id, roles}
     → PermissionInterceptor:
         1. 查路由对应权限码（注解 / 静态表）
         2. 检查 user → roles → permissions 集合是否含该码
         3. 若涉及具体资源：再查 ACL 覆盖
         4. 跨租户访问：拒绝（除 super_admin）
     → 业务 handler
```

权限注解（Go 端用静态映射）：

```go
// admin-api/internal/auth/permissions.go
var rpcPermissions = map[string]string{
    "/alfq.v1.StrategyService/DeployStrategy": "strategy.deploy",
    "/alfq.v1.RiskService/KillSwitch":         "risk.kill_switch",
    // ...
}
```

### 2.6 前端鉴权

- JWT payload 含 `permissions[]`，前端组件级隐藏不可见操作
- 路由守卫双重检查
- **永远以后端为准**，前端隐藏只是 UX

## 3. 认证

### 3.1 流程

```
[1] Login(email,password,tenant)
     → 校验 + 检查 TOTP enabled
       - 未启用 TOTP：直接返回 access+refresh
       - 已启用：返回 totp_challenge
[2] VerifyTOTP(challenge, code) → 返回 access+refresh
[3] 携带 Bearer 访问 RPC
[4] access 过期 → Refresh(refresh_token) → 新对
[5] Logout → 撤销 refresh + JWT jti 加入黑名单
```

### 3.2 Token

- **Access JWT**：HS256/EdDSA（推荐 EdDSA），15 min，含 `sub/tid/roles/perms/jti/exp/iat`
- **Refresh Token**：随机 256-bit，**sha256** 哈希后存 PG（参见 doc 22 §2 PII 矩阵），7 天，可滚动续期
- **存储**：
  - Access：浏览器内存
  - Refresh：**HttpOnly + Secure + SameSite=Strict cookie**，path=`/api/auth/refresh`
- **撤销**：登出/改密/异常 → 黑名单到 Redis（TTL=剩余 exp）

### 3.3 TOTP 二次验证

- 启用：`EnrollTOTP` 返回 otpauth URL 与二维码，用户扫码并验证一次确认
- secret 用 Vault Transit 加密存 PG（`users.totp_secret`）
- 敏感操作（如 KillSwitch / DeployStrategy）即使已登录也需当次再输入 TOTP

### 3.4 密码策略

- argon2id（内存 64MB / 时间 3 / 并行 1）
- 最小 12 位 + 复杂度（zxcvbn 得分 ≥ 3）
- 不在常见泄露库（Have I Been Pwned k-anon API，可选）
- 失败 5 次锁 15 分钟（Redis 计数）

### 3.5 SSO（v2）

预留 OIDC 接入（Keycloak / Auth0），第一期不实现。

## 4. 与 Notebook 的集成

JupyterHub 用 OAuth 与 `admin-api` 对接：

```
浏览器 → /notebooks → admin-api 签发 短期 OIDC code → JupyterHub
JupyterHub → 创建沙箱容器（per-user），注入：
   - 用户 PG 只读用户名/密码（限定 RLS 上下文）
   - 用户 CH 只读账号（限定 row policy）
   - MinIO 临时 STS 凭据（限定前缀 tenants/{tid}/notebooks/{uid}/）
   - JWT（用于调 admin-api）
```

容器 lifetime ≤ 8 小时，空闲 30 分钟回收。

## 5. 跨租户操作（super_admin）

- 必须显式带 `X-Tenant-Id` header 才能切换上下文
- 每次跨租户操作强制写审计 + 告警
- 提供"模拟用户登录"功能（impersonate），同样审计

## 6. 审计

- 所有写 RPC 自动写 `audit_logs`（02 章表）
- 关键字段：actor、action、resource、payload(脱敏)、ip、result
- 审计表 INSERT ONLY，触发器拒绝 UPDATE/DELETE
- 每日异步归档到对象存储（Object Lock 不可删）
