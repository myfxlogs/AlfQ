## 2. 技术决策汇总

### 2.1 通信协议

| 场景 | 协议 | 库 |
|---|---|---|
| 浏览器 ↔ Admin API | **Connect (HTTP/JSON 或 Connect-Web)** | `@connectrpc/connect-web` + `connect-go` |
| 内部 Go 服务之间 | **gRPC over HTTP/2** | `connect-go`（同源） |
| Go ↔ mtapi 网关 | **gRPC over TLS**（远端 MT4=`mt4grpc3.mtapi.io:443` / MT5=`mt5grpc3.mtapi.io:443`；**两套独立 proto**：`gprc/mt4.proto`、`gprc/mt5.proto`，不可共用类型，详见 doc 14 §3.4 与 doc 25 §3） | `google.golang.org/grpc` |
| 服务器推送 | **Server Streaming RPC**（Connect 上是 SSE 传输） | 同上 |
| 异步消息 | **NATS JetStream** | `nats.go` |
| 健康检查 | HTTP `/healthz` `/readyz`（仅此 REST） | `net/http` |

**Connect 一份 .proto 同时出**：gRPC、gRPC-Web、Connect 三种协议端点。前端直接用 TS 类型。

#### 2.1.1 序列化格式边界（JSON vs Protobuf）

**总原则**：跨进程契约用 protobuf（单一源 `backend/proto/alfq/v1/`），进程内 / 工具链交互用 JSON / YAML。详见 doc 25 §6（如有）与下表：

| 场景 | 格式 | 理由 |
|---|---|---|
| RPC（服务间 / 前后端） | **protobuf** | 强类型、版本演进、跨语言 |
| NATS JetStream 消息体 | **protobuf**（`outbox.payload BYTEA`） | 跨服务契约统一，详见 doc 21 §5.2、§5.6 |
| SSE / Server Streaming 推送 | **protobuf**（Connect 自动协商） | 与 RPC 同源 |
| **结构化日志**（zap → Loki） | **JSON** | Loki/Promtail 链路只解析 JSON/logfmt；运维 `grep`/`jq`；二进制不可读 |
| 配置文件（trading-core.yaml 等） | **YAML** | 人工编辑、Git diff、注释支持 |
| DB JSONB 字段（`metadata`、`audit.detail`、`saga.steps`、`outbox.headers`） | **JSON** | PG GIN 索引、`->>` 查询、灵活演进 |
| HTTP `/healthz` `/readyz` | **JSON** | k8s probe / 监控工具链标准 |
| AI Agent tool I/O / 因子 DSL | **JSON / 字符串** | LLM 原生输出、人工调试 |
| 测试夹具 / mock fixture | **JSON / YAML** | 可读、易改 |

**强一致来源**：跨语言离散值（错误码、订单状态、品种类型、`position_mode` 等枚举）以 `backend/proto/alfq/v1/` 中的 `enum` 为唯一源，由 codegen 反哺 Go 常量、TS 类型、SQL CHECK 约束。

### 2.2 数据存储

| 用途 | 选型 | 备注 |
|---|---|---|
| 交易主数据（账户/订单/用户/策略元数据/审计） | **PostgreSQL 16** | 强一致、RLS 多租户 |
| 时序数据（tick/bar/因子/信号/成交流水） | **ClickHouse 24+** | 按 `tenant_id`+月份分区 |
| 缓存/会话/分布式锁/限流 | **Redis 7** | + RedisJSON 可选 |
| 对象存储（模型/报告/快照/归档） | **MinIO（dev）/ S3（prod）** | |
| 秘钥/凭据 | **HashiCorp Vault** | 动态凭据 + Transit 加密 |
| 消息总线 | **NATS JetStream** | 持久化 + at-least-once |

### 2.3 服务划分

| 服务 | 语言 | 端口 | 职责 |
|---|---|---|---|
| `trading-core` | Go | 9000 | 对外 API（Connect+SSE）+ 订单管理 + 风控（合并 admin-api + oms + risk-svc） |
| `md-gateway` | Go | 9001 | mtapi 行情接入 → NATS |
| `quant-engine` | Go | 9002 | 因子计算 + 策略执行（合并 factor-svc + strategy-svc） |

### 2.4 用户策略执行模型（核心安全决策）

**禁止**用户编写的 Python 代码进入生产实盘进程。生产策略只能由以下两类构成：

1. **因子 DSL 字符串**：用户在 UI 或 Notebook 中编辑表达式，平台审核入库
2. **ONNX 模型文件**：研究层训练后导出，由 `quant-engine` 用 Go ONNX Runtime 加载

研究层 Python 代码运行在**隔离沙箱容器**中（详见 `06-Python策略沙箱设计.md`），用于：
- 数据探索 / 特征工程 / 模型训练 / 回测
- 沙箱无外网（白名单 ClickHouse/PG/对象存储）
- 资源配额（CPU/内存/时间），异常 OOM kill

平台开发者维护的 Python 算子库（如新增 DSL 算子）走代码评审 + CI 进入主仓库，与用户代码隔离。

### 2.5 多租户模型

| 层 | 隔离方式 |
|---|---|
| 应用层 | 所有请求 JWT 携带 `tenant_id`，gRPC interceptor 注入 context |
| PG 表 | 每张业务表带 `tenant_id` 列 + Row-Level Security 策略 |
| CH 表 | 每张时序表 `ORDER BY (tenant_id, symbol, ts)`，查询必带 `tenant_id` 谓词 |
| 经纪商账号 | **物理隔离**：每租户独立的 mtapi 连接池与账号凭据 |
| Notebook | 每用户独立沙箱容器、独立卷、独立 ClickHouse user（限定 row policy） |
| 对象存储 | 桶按 `tenant/{tenant_id}/...` 前缀分区，IAM 策略限定 |

### 2.6 权限模型（RBAC + ACL）

角色分两类，可叠加授予同一用户。

**A. 业务角色**（系统内置，必备）：
- `super_admin`：平台级，跨租户
- `tenant_admin`：租户管理员
- `trader`：可下单、查看自己策略
- `quant`：可写因子/策略/回测，不可触发实盘
- `viewer`：只读

**B. 治理/合规角色**（可选，用于敏感评审与值班）：
- `risk_officer`：风控规则/资金路径变更评审、KillSwitch 双人审批
- `security_officer`：安全相关 ADR/事件评审
- `privacy_officer`：数据/PII/GDPR 主体权利评审
- `legal_officer`：法务相关合规评审

业务角色决定**日常操作权限**；治理角色决定**变更与事件评审权限**（详见 doc 17 §6.1、doc 19 §6、doc 22 §6）。

**资源 ACL**：策略、账户、因子可单独授予用户读/写/执行权限，覆盖角色默认。

**敏感操作 TOTP 二次验证**：实盘启停、Kill Switch、提取凭据、修改风控阈值。

### 2.7 部署形态

- **dev**：单机 docker-compose（含所有依赖）
- **staging**：单机 docker-compose（4C/8G）
- **prod**：单机 docker-compose（16C/32G/SSD），详见 ADR 0011

## 5. 关键非功能需求（NFR）

> **NFR / SLO / SLA 关系**（三者必须区分）：
> - **NFR（本表）**：设计目标。架构必须支持，工程基线。
> - **内部 SLO**（doc 15 §6.1）：实测保守值，用于错误预算与发布门禁，**比 NFR 宽松**。
> - **对外 SLA**（doc 24 §1.1）：商业承诺，按 plan 分层（Basic/Pro/Enterprise），**比 SLO 宽松**。
> - 关系：**NFR（设计目标） ≥ 内部 SLO（实测）≥ 对外 SLA（承诺）**。

| 指标 | NFR 设计目标 | 内部 SLO（doc 15） | 对外 SLA Enterprise（doc 24） |
|---|---|---|---|
| 前端首屏 | < 2s（CDN + 代码分割） | — | — |
| Admin API p99 | ≤ 200 ms（普通查询）/ ≤ 100 ms（核心读） | < 200 ms | — |
| 内部 gRPC 单跳 p99 | ≤ 5 ms | — | — |
| 行情 tick → 策略 p99（本地侧） | ≤ 20 ms | — | — |
| 端到端下单 p99（含 broker） | ≤ 150 ms | < 500 ms（含 broker 波动） | — |
| 服务可用性 | ≥ 99.95% | ≥ 99.9% | 99.95% |
| RPO（业务主数据/订单） | ≤ 1 分钟（Outbox + WAL 流复制） | — | — |
| RPO（PG 全量恢复） | ≤ 5 分钟 | — | — |
| RTO（同区域故障切换） | ≤ 30 分钟 | — | — |
| RTO（跨区域容灾） | ≤ 2 小时 | — | — |

详见：doc 11 附录 B（DR 演练）、doc 15 §6（SLO 表）、doc 24 §1（SLA 表）。
