# AI Agent 执行 Runbook（唯一入口）

> **本文是新接手的 AI Agent 进入仓库后看到的第一份文件**。
> 当你（AI Agent）被分配到 ALFQ 仓库时：**先读完本文 → 再去做事**。
>
> 维护规则：本文只描述"怎么干活"。设计、规范、里程碑细节都不写在这里，只放链接。

## 2. 执行循环（每个任务都跑一次）

```
┌──────────────────────────────────────────────────────────────┐
│  STEP A · 选任务                                             │
│   打开 docs/tasks/MASTER-ROADMAP.md §11，                   │
│   选当前阶段第一个状态为 ☐ 的任务                            │
│   （不允许跨阶段跳取；不允许跳过未完成的依赖）              │
└──────────────────────────────────────────────────────────────┘
                              ↓
┌──────────────────────────────────────────────────────────────┐
│  STEP B · 上下文                                             │
│   按本文 §1 "额外按需阅读" 表，根据任务标题决定要读哪些     │
│   读完后写一段 ≤ 200 字的 "我准备这样做" 简报到 chat        │
│   （不要直接动手，先确认理解正确）                          │
└──────────────────────────────────────────────────────────────┘
                              ↓
┌──────────────────────────────────────────────────────────────┐
│  STEP C · 编码                                               │
│   严格按该任务 "改动范围" 列出的文件清单动手                │
│   不允许扩张范围（如确实需要 → 回到 STEP B 改简报，停车）   │
│   遵守 AGENT.md 的复杂度硬上限                              │
│   每写完一个文件运行一次 lint / build                       │
└──────────────────────────────────────────────────────────────┘
                              ↓
┌──────────────────────────────────────────────────────────────┐
│  STEP D · 验收                                               │
│   照搬任务节里的 "验收" 命令，逐条跑                        │
│   全部通过才算完成。不通过 → 回 STEP C 修复，不调低标准    │
└──────────────────────────────────────────────────────────────┘
                              ↓
┌──────────────────────────────────────────────────────────────┐
│  STEP E · 提交 + 收尾                                        │
│   按 §4 提交 PR                                              │
│   在 docs/tasks/MASTER-ROADMAP.md §11 把 ☐ 改为 ☑          │
│   在 §12 工作日志追加一行（日期 / Agent / 任务 ID / 简报）   │
└──────────────────────────────────────────────────────────────┘
```

## 3. 卡住怎么办

按以下决策树**严格执行**，不要发挥：

| 情形 | 处置 |
|---|---|
| 某文档与另一文档冲突 | 查 `AGENT.md §"文档唯一源"` 表，按那里指定的为准；冲突情况写到 PR description |
| 任务的"改动范围"实际不够 | **不**自行扩张。在 chat 中报告："任务 X 的范围需要扩展到 Y，建议拆 PR 或修订 ROADMAP" |
| 任务依赖的 P0 项还没完成 | **不**绕过。回到 §2 STEP A 选别的任务，或报告无可选任务 |
| 验收命令跑不通且非代码 bug（环境问题） | 在 PR description 提供复现步骤，标记 "blocked: env"，**不勾选**完成 |
| 发现 ROADMAP 有错 | 在 chat 报告并提议修订；不要在自己的 PR 里偷偷改 ROADMAP |
| 发现 ADR 与代码冲突 | 立即停手，报告人类负责人决定改 ADR 还是改代码 |
| 上下文窗口要满了 | 立刻总结当前进度到 chat，请人类决定继续 / 换 Agent / 拆任务 |
| 任何安全 / 合规疑问 | 立即停手，不凭常识决定。引用 AGENT.md 红线条款 |
| 之前已勾 ☑ 的任务被发现实际未在真数据上验证 | 不要悄悄改回 ☐。在 §12 工作日志显式记录"伪完成"，状态改为 🅒（code-only），并新建 `<TaskID>.<n>` 修复任务挂在 §11.1 |
| 任务声称"完成"但跑不通 ROADMAP 的"验收"命令 | 任务状态保持 ☐ 或 🅒；在 chat 报告"验收命令 X 失败，原因 Y"；不允许跳过、不允许降低验收标准 |
| 任务的 fixture / endpoint / 数据 不存在 | 在 ROADMAP §11.1 该任务后加 `blocked: <reason>`；先做不依赖该资源的任务；不允许伪造 fixture |
| 看到 🅒 状态的任务想升级到 ☑ | 必须先确认 DEP-1 已解锁、真实数据可用，再跑该任务节"验收"命令全部通过；任一不满足保持 🅒 |

**绝对禁止**：
- 跳过验收命令
- 修改测试以使其通过
- 用 `// nolint` 绕过 lint
- 凭直觉决定 MT4/MT5 接口能否复用（必须查 `docs/29`）
- 跨阶段并行执行任务

## 4. 提交规范（与 AGENT.md 一致，这里给最小操作集）

### 4.1 Commit message

```
<type>(<scope>): <subject>     ← 主题 ≤ 60 字

<body>                          ← 可选，说明 why
                                 ← 关联文档 / PR 链接

Refs: docs/tasks/MASTER-ROADMAP.md §<N> (任务ID)
```

`type` 取值：`feat | fix | chore | docs | refactor | test`
`scope` 取值：`mdgateway | symbolsync | factorsvc | strategysvc | quantengine | oms | risksvc | tradingcore | research | frontend | proto | deploy | docs`

### 4.2 PR 必备清单

- [ ] commit message 符合 §4.1
- [ ] 关联到 MASTER-ROADMAP 任务 ID
- [ ] 改动文件全部在该任务 "改动范围" 内
- [ ] 单测覆盖率达标（关键路径 ≥ 60%）
- [ ] 验收命令复制粘贴 + 执行结果截屏 / 日志
- [ ] 复杂度上限合规（参见 AGENT.md）
- [ ] 涉及架构 → 附 ADR
- [ ] 涉及 MT4/MT5 → PR description 引用 `docs/29` 对应章节
- [ ] 涉及 proto → 跑过 `buf lint` + `buf breaking`

### 4.3 PR description 模板

```markdown
## 任务
MASTER-ROADMAP: <Task ID>（如 DP-1）

## 改动摘要
- 一句话描述做了什么

## 关联文档
- docs/tasks/MASTER-ROADMAP.md §X
- docs/<N>-...md §X（如有）

## 验收
\```bash
# 复制 ROADMAP 任务里的"验收"命令
\```
结果：（贴执行输出）

## 风险与回滚
- 风险：...
- 回滚：git revert <commit>，无 schema 变更 / 含 schema 变更（描述兜底）

## 复杂度
- 单文件最大 X 行 / 单函数最大 X 行 / 圈复杂度 X
```

## 5. 阶段收尾

完成一个 Phase 全部任务后：

1. 在 `docs/handover/` 新建 `M<X>.<Y>-<scope>-handover.md`，结构参考最近一份
2. 在 MASTER-ROADMAP §11 进度表确认全部 ☑
3. 在 §12 工作日志写一段"阶段总结"
4. 通知人类，等待 Phase 切换指令
5. **不要**自行启动下一 Phase（即使 ROADMAP 里看似允许）

## 6. 常见陷阱（来自历史 Agent 失败案例）

| 陷阱 | 表现 | 防御 |
|---|---|---|
| 文档崇拜 | 把全部 28 份 doc 都读一遍，上下文耗尽 | 严格按 §1 表读，不在表里的等触发再读 |
| 范围爆炸 | "顺手"把另一个 bug 一起改了 | 严格按"改动范围"清单；额外发现的 bug 写到 §3 报告 |
| 验收降级 | 验收命令跑不过就改测试或注释 | 验收失败 = 任务未完成，回 STEP C |
| 跨阶段提前 | Phase A 没全完就动 Phase C | §2 STEP A 不允许跨阶段 |
| 平台混淆 | MT4 fetcher 套 MT5 字段路径 | 强制读 docs/29，每个字段注释来源 |
| 凭直觉决合规 | 安全/法务问题自行判断 | §3 末"凭常识"禁条 |
| 漂移升级 | 自己悄悄改 ROADMAP 或 ADR | §3 "发现错"流程，先报告再改 |
| 上下文丢失 | 跨多个 PR 后忘了之前怎么做的 | §5 强制 handover；不省略日志 |

## 8. 与现有文档的关系

```
AGENT.md                    项目身份 / 红线 / 不可变规则
  │
  ├─→ docs/README.md        完整文档地图（工具书）
  │
  ├─→ docs/12-...           编码 / 提交 / 复杂度细则
  │
  └─→ docs/tasks/AGENT-RUNBOOK.md (本文)    ← AI Agent 唯一执行入口
        │
        └─→ docs/tasks/MASTER-ROADMAP.md   ← 当前任务清单
              │
              ├─→ docs/tasks/QUANT-EXECUTION-PLAN.md     (附录-研究细节)
              ├─→ docs/tasks/DATA-INFRA-AUDIT-2026-05-20.md (附录-数据底座诊断)

```

**优先级**：
- AGENT.md > 任何其它文档
- AGENT-RUNBOOK.md（本文）> MASTER-ROADMAP.md（操作规则 > 任务表）
- MASTER-ROADMAP.md > 个别 doc（任务范围 > 一般规范）
- 个别 doc > 任何外部知识（项目规范 > 业界常识）

冲突时按上述优先级解决。
