# 23 - 限流、熔断与配额

> 所有外部入口、内部依赖、租户资源都必须有限流与熔断；所有 SaaS 资源都必须有配额。本文给出**具体桶大小、阈值、降级策略**。

## 1. 多层防护体系

```
[Edge: Caddy/Nginx]      ─ IP 级 + Bot 防护
       │
[Admin API: Interceptor] ─ User / Tenant 级 + Per-RPC
       │
[Internal Services]       ─ Circuit Breaker + Bulkhead
       │
[Dependencies: PG/CH/Redis/NATS/mtapi] ─ 连接池上限 + Timeout
```

## 2. Edge 层（Caddy）

每 IP 限制：

| 路径 | 限制 |
|---|---|
| `/api/auth/login` | 5 req / 1 min |
| `/api/auth/refresh` | 30 req / 1 min |
| `/api/*` 其他 | 300 req / 1 min |
| 流式 `/api/*.Stream*` | 30 连接 / 同 IP |

超限返回 `429 Too Many Requests` + `Retry-After`。

Bot 防护：UA 黑名单、可疑 path 拒绝、CAPTCHA（敏感操作）。

## 3. Admin API 层

### 3.1 限流维度

按以下复合 key 限流（token bucket，Redis 实现）：

```
{tenant_id}:{user_id}:{rpc_method}
```

| 类别 | 默认（per minute） | 备注 |
|---|---|---|
| **读** | | |
| Get/List 通用 | 600 | |
| StreamQuotes / Bars | 10 连接同时 | |
| **写** | | |
| PlaceOrder | 60 | 单用户每秒 1 笔 |
| CancelOrder | 120 | |
| DeployStrategy | 10 | |
| KillSwitch | 5 | + TOTP |
| EnrollTOTP / Login | 10 | + IP 限流 |
| AI 助手对话 | 30 | + tenant 日 token 配额 |
| BackTest start | 20 | |

### 3.2 实现

`backend/go/internal/common/ratelimit/`：

```go
type Limiter interface {
    Allow(ctx, key, capacity, refillRate) (allowed bool, retryAfter time.Duration, err error)
}
// 实现：Redis token bucket（GCRA / Lua atomic）
```

Connect interceptor 自动按 RPC method 查配置应用。

### 3.3 配置文件

`configs/ratelimit.yaml`，可热加载：

```yaml
defaults:
  burst: 1.5            # bucket = rate * burst
rules:
  - rpc: alfq.v1.OrderService/PlaceOrder
    per: user
    rate: 60
    window: 1m
  - rpc: alfq.v1.AuthService/Login
    per: ip
    rate: 5
    window: 1m
  - rpc: alfq.v1.RiskService/KillSwitch
    per: user
    rate: 5
    window: 1m
overrides:
  tenants:
    "premium-tenant-uuid":
      rate_multiplier: 3
```

### 3.4 限流响应

- Connect Code: `ResourceExhausted`
- ErrCode: `ERR_RATE_LIMITED`
- Header: `Retry-After: <seconds>`
- 不计入错误率 SLO

## 4. 租户配额

### 4.1 模型

`tenants.plan` 决定（plan 用 enum 字符串）：

| Plan | Basic | Pro | Enterprise |
|---|---|---|---|
| max_users | 5 | 25 | 100 |
| max_accounts | 3 | 10 | 50 |
| max_strategies | 20 | 100 | 500 |
| max_deployments_concurrent | 5 | 20 | 100 |
| max_factors | 50 | 200 | 1000 |
| max_backtests_per_day | 50 | 500 | 5000 |
| ai_daily_tokens | 100k | 1M | 10M |
| data_retention_days | 90 | 365 | 730 |
| notebook_cpu | 1 | 4 | 16 |
| notebook_mem_gb | 2 | 8 | 32 |
| api_req_per_min | 1000 | 5000 | 50000 |
| stream_connections | 10 | 50 | 500 |

存 `plans` 表 + `tenants.plan_overrides JSONB`（特殊客户）。

### 4.2 配额检查

每个写 RPC 进入业务逻辑前调 `quota.Check(ctx, tenant_id, resource_type)`：

- 创建用户 → check `users` 数量
- 创建策略 → check `strategies` 数量
- 启动回测 → check `daily_backtests`
- AI 调用 → check `daily_tokens`

超限：`ERR_QUOTA_TENANT_LIMIT` / `ERR_AI_DAILY_TOKEN_EXCEEDED`。

### 4.3 配额表

```sql
CREATE TABLE tenant_quota_usage (
  tenant_id    UUID NOT NULL,
  resource     TEXT NOT NULL,
  period       TEXT NOT NULL,        -- 'daily' / 'total' / 'concurrent'
  bucket       DATE,                  -- 日期 bucket（daily 用）
  used         BIGINT NOT NULL DEFAULT 0,
  PRIMARY KEY (tenant_id, resource, period, bucket)
);
```

每日凌晨重置 daily；concurrent 在创建/释放时增减。

### 4.4 计费预留（toB SaaS）

虽然首期自营无外部计费，但模型支持：

- `billing_events` 表记录每次资源使用（用户/策略/AI tokens/Backtest）
- 定期聚合输出 CSV / 接入第三方计费（Stripe 等）

```sql
CREATE TABLE billing_events (
  id          BIGSERIAL PRIMARY KEY,
  tenant_id   UUID NOT NULL,
  kind        TEXT NOT NULL,        -- ai_tokens / backtest_minutes / strategy_active_days / ...
  units       NUMERIC(20,4),
  meta        JSONB,
  occurred_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

## 5. 熔断器（Circuit Breaker）

### 5.1 适用场景

- 调下游服务 / 外部 API（mtapi / LLM provider）
- DB 连接池满
- 行情订阅异常累积

### 5.2 实现

`backend/go/internal/common/breaker/`：

库：[sony/gobreaker](https://github.com/sony/gobreaker) 或自研三态状态机。

```go
type Breaker struct {
    Name           string
    MaxRequests    uint32       // half-open 允许的探测请求
    Interval       time.Duration // closed → reset 计数
    Timeout        time.Duration // open → half-open
    FailureRatio   float64       // 触发 open 的失败率
    MinRequests    uint32        // 评估窗口最少请求数
}
```

### 5.3 默认配置表

| 目标 | failure_ratio | min_req | timeout |
|---|---|---|---|
| mtapi 单 broker 连接 | 0.5 | 20 | 30s |
| LLM provider | 0.3 | 10 | 60s |
| PG 单实例 | 0.5 | 50 | 10s |
| CH | 0.5 | 50 | 30s |
| Redis | 0.7 | 100 | 5s |
| 内部 RPC | 0.5 | 50 | 10s |

### 5.4 半开降级

熔断打开期间：

| 场景 | 降级策略 |
|---|---|
| mtapi 断 | 拒绝新单（保护资金），现有部署 paused |
| LLM 断 | AI 助手返回 ERR_AI_PROVIDER_ERROR + 提示稍后再试 |
| Redis 断 | 关掉缓存层走 DB；幂等改用 DB 唯一约束 |
| CH 写入断 | 暂存本地磁盘 outbox-style 队列 |
| PG 读断 | 切只读副本（若有） |

熔断状态指标：`alfq_breaker_state{scope}` → 触发告警。

## 6. Bulkhead（舱壁隔离）

避免单租户拖垮：

- 每租户独立 goroutine pool（订单处理 / 流式订阅）
- DB 连接池按租户加 weight（高级 plan 多分配）
- AI 助手 LLM 调用按租户独立 semaphore

`common/bulkhead/`：

```go
b := bulkhead.New(name="ai-llm", max=50, perTenant=5)
b.Do(ctx, tenantID, func() error { ... })
```

## 7. 超时

所有 IO 都必须有超时（无默认无超时）：

| 调用 | timeout |
|---|---|
| Connect/gRPC server handler | 由 ctx 控制，默认 30s |
| Connect/gRPC client | 操作类型定（读 5s / 写 10s / 流 无限） |
| PG query | 普通 3s，分析 30s |
| CH query | 普通 5s，分析 60s |
| Redis op | 1s |
| NATS publish | 2s |
| mtapi RPC | 5s |
| LLM call | 60s |
| HTTP 外部 | 10s |

`common/timeout/policies.go` 集中管理。

## 8. 流式连接限制

- 单服务实例 SSE 最大并发：1 万
- 单用户 SSE 最大并发：10
- 单 tenant SSE 最大并发：按 plan
- 心跳：每 30s server 发 ping；客户端 60s 无消息断开
- 闲置超时：5 分钟无订阅活动断开

## 9. 监控

- `alfq_rate_limit_throttle_total{rpc, tenant_bucket}` ← Prom
- `alfq_quota_used / alfq_quota_limit{resource, tenant}` ← Prom
- `alfq_breaker_state{scope}` ← Prom
- `alfq_concurrent_streams{service}` ← Prom

告警：
- 限流命中率 > 5% → warning
- 配额使用 > 80% → 通知 tenant_admin
- 熔断 open > 1 分钟 → critical
