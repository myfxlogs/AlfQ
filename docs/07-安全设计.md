# 07 - 安全设计

## 1. 威胁模型（STRIDE 摘要）

| 威胁 | 关注点 | 主要对策 |
|---|---|---|
| Spoofing | 冒充用户/服务 | JWT + TOTP + mTLS（内部）|
| Tampering | 篡改订单/参数 | 审计 + 签名 + DB 行锁 + Idempotency |
| Repudiation | 操作抵赖 | 不可变审计日志 + 对象锁归档 |
| Info Disclosure | 凭据/资金信息泄露 | Vault + TLS + 字段加密 + RBAC + RLS |
| DoS | 资源耗尽 | 限流 + 配额 + 熔断 |
| Elevation of Privilege | 横向/纵向提权 | 最小权限 + 二次验证 + 审计告警 |

## 2. 秘钥与凭据管理

### 2.1 Vault

- HashiCorp Vault（生产 HA 模式），单租户/多租户独立路径
- 启用：
  - **KV v2**：第三方 API Key（经纪商账号密码因 MT 协议要求明文递交上游，存于 PG 磁盘加密 + RLS）
  - **Transit**：字段加密（如 PG 中 `totp_secret`）
  - **PKI**：内部服务 mTLS 证书签发
  - **Database**：动态生成 PG/CH 短期凭据
- 应用以 **Vault Agent sidecar** 注入秘钥到内存或 tmpfs，绝不写入镜像

### 2.2 路径约定

```
kv/system/jwt/{kid}                 # JWT 签名密钥（轮换）
kv/system/mtapi/auth
kv/tenants/{tid}/accounts/{aid}     # ~~broker 账号密码~~（已改为 PG 明文存储，见 doc 02 §1.3）
transit/keys/users-totp             # TOTP secret 加密 key
pki/issue/internal-services         # 内部 mTLS
```

### 2.3 轮换

| 秘钥 | 周期 |
|---|---|
| JWT 签名 key | 90 天，支持多 kid 并行 |
| 内部服务证书 | 30 天自动续 |
| Broker 账号密码 | 用户可手动改，平台不主动 |
| Vault root token | 仅初始化用，之后销毁 |

## 3. 传输安全

- **外部**：HTTPS only，TLS 1.3，HSTS，CSP 严格
- **内部**：mTLS（PKI 签发），SPIFFE ID 形式 `spiffe://alfq.internal/svc/<name>`
- WAF：Caddy / Cloudflare / 自建（OWASP CRS）
- 速率限制：Edge 层（每 IP）+ App 层（每用户/租户）

## 4. 应用安全

### 4.1 输入校验

- 所有 RPC 参数走 protobuf 校验 + buf 的 `protovalidate` 规则
- DSL 表达式：白名单 token，AST 校验，禁止递归/网络/IO 调用
- SQL：100% 参数化查询，禁字符串拼接
- 文件上传：MIME + 后缀 + 魔数三重检查；ONNX 文件校验 model proto 头

### 4.2 输出/序列化

- JSON：标准库默认安全
- HTML 报告：服务端生成、CSP 隔离 iframe 嵌入；DOMPurify 二次净化

### 4.3 依赖

- Go：`govulncheck` + `osv-scanner` CI 必跑
- Python：`pip-audit` + `bandit`
- TS：`pnpm audit` + Snyk
- 镜像：Trivy 扫描，禁止 CVE high+

### 4.4 SBOM

每镜像产出 SPDX，上传到工件库。

## 5. 数据安全

### 5.1 静态加密

- 数据库磁盘加密（云：默认；自建：LUKS）
- 关键列字段级加密（Vault Transit）：`users.totp_secret`
- `accounts.password` 因 MT 协议要求明文递交上游交易服务器，不做应用层加密，依赖 PG 磁盘加密 + RLS + 最小权限 + 审计补偿
- 对象存储桶 SSE-KMS

### 5.2 传输加密

见 §3。

### 5.3 PII / 合规

- 邮箱 / IP / 用户行为为 PII，访问需 `audit.read` 权限
- 删除用户 → 个人数据 30 天软删 → 物理清除（审计日志匿名化保留）
- 日志中禁止打印 password / token / 模型文件 / 完整 JWT；做 `*` 脱敏

## 6. 审计与不可篡改

- `audit_logs` 表 + 触发器禁 UPDATE/DELETE
- 每日把当日日志归档到对象存储，启用 **Object Lock**（compliance mode）保留 5 年
- 审计日志哈希链可选（每行附 prev_hash），用于发现篡改

## 7. 风控/Kill Switch 安全

- KillSwitch RPC 需要：
  1. 当前 user 拥有 `risk.kill_switch` 权限
  2. 即时 TOTP 校验
  3. 输入 confirm phrase
  4. 双人模式（可配置）：另一个有权角色二次批准（10 分钟超时）
- 触发后所有部署状态置 `halted`，broker 连接保留以便撤单/平仓
- 解除需 `tenant_admin` 手动操作

## 8. 部署安全

### 8.1 容器

- 镜像非 root；`USER 65534`
- read-only root FS
- `seccomp=RuntimeDefault` + 必要 capabilities 显式 drop ALL
- `securityContext.allowPrivilegeEscalation=false`

### 8.2 K8s

- NetworkPolicy 默认 deny，按服务白名单
- ServiceAccount 最小 RBAC
- Secrets 从 Vault 注入，禁用集群明文 Secret（或加密 etcd）
- PSP/PodSecurity `restricted`

### 8.3 主机

- VPS 仅开 22/443/必要内网端口
- SSH key only，禁密码，fail2ban
- 内核更新自动化（unattended-upgrades）
- 日志远程发送 Loki，禁止本地修改 root 日志

## 9. CI/CD 安全

- 仓库保护分支，需双人审批
- 提交签名（GPG / sigstore）
- 流水线最小权限 token（GitHub OIDC → 云 IAM）
- 制品签名（cosign）+ admission controller 校验
- Secrets 不入仓库（pre-commit gitleaks）

## 10. 监控告警

- 安全相关告警 P0：
  - Vault sealed
  - mTLS 证书即将过期
  - 审计日志写失败
  - 异常登录（地理跳变、撞库特征）
  - 跨租户访问尝试
  - KillSwitch 触发
  - CH/PG 大量失败查询

## 11. 应急响应

提供 Runbook（`docs/runbook/incident-*.md`）：

- 凭据泄露 → 撤销 / 轮换 / 通告
- Broker 资金异常 → 立即 KillSwitch + 锁账户
- 服务被攻击 → 限流 / 切流 / 上 WAF 规则
- 数据库故障 → 切只读 / 切只读副本 / 拒新单
- 重大 Bug 致错单 → 撤单脚本 + 客户通告

## 12. 合规

- 数据保留：交易 ≥ 5 年，审计 ≥ 5 年，登录日志 ≥ 1 年
- KYC：第一期为自营，不收集客户资金；面向外部前需补 KYC/AML
- 数据所在地：按租户配置（GDPR 可选 EU region）

## 13. 验收检查项

- [ ] Vault 集成测试（启动加载 / sidecar 注入）
- [ ] mTLS 内部通信测试（自签链）
- [ ] 渗透测试关键路径：登录 / 越权 / 注入 / XSS / CSRF
- [ ] 容器 root 运行被拒（pod security policy）
- [ ] 跨租户访问被拒（自动化测试）
- [ ] 审计 UPDATE/DELETE 触发器测试
- [ ] KillSwitch 端到端测试
- [ ] 漏洞扫描 CI 通过、镜像签名校验通过
