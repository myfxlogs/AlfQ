## 2. 发现的冲突（9 类）

### A. NFR / SLO / SLA 数字不一致【严重】

涉及：doc 01 §5、doc 11 附录 B、doc 15 §6、doc 16 §7、doc 24 §1、落地方案 §11

| 冲突点 | doc 01（修复前） | 各处实际 |
|---|---|---|
| 可用性 | 99.95% | 15: 99.9% / 24: 99.5/99.9/99.95% 三档 |
| Admin API p99 | < 100 ms | 15 SLO: < 200 ms / 16 bench: < 100 ms |
| RPO | < 1 分钟 | 11 附录 B: PG 5 分钟 |
| RTO | < 5 分钟 | 11 附录 B: PG 30 分钟 |
| 端到端下单 p99 | < 150 ms | 15: < 500 ms（含 broker 波动）|

**根因**：未区分"设计目标 / 实测保守值 / 商业承诺"三者关系。

### B. 订单状态机 5 个版本【严重】

**根因**：状态机在 doc 14 才完整化，前序文档未回溯。

### C. assistant-svc 端口 9006 未进 doc 01 服务表【中】

doc 18 §B2.1 新增 `assistant-svc:9006`，但 doc 01 §2.3 服务表未加，会让 AI Agent 看 doc 01 时遗漏。

### D. 14 张新表未在 doc 02 索引【中】

后续文档新增表：`trading_calendar / feature_flags / idempotency_records / sagas / outbox / ai_conversations / ai_messages / ai_artifacts / ai_quotas / models / model_drift_metrics / user_consents / data_lineage / plans / tenant_quota_usage / billing_events / support_tickets / support_messages`，doc 02 无入口，Agent 易遗漏。

### E. RBAC 角色清单不完整【中】

doc 01 §2.6 只列 5 个基础角色，但 doc 17 §6.1 / doc 19 §6 / doc 22 §6 用到 `risk_officer / security_officer / privacy_officer / legal_officer` 评审，doc 01 未声明。

### F. Refresh token 哈希算法不一致【小】

- doc 05 §3.2："bcrypt/sha256"
- doc 22 §2 PII 矩阵："sha256"

**正确选择**：sha256（token 已是 256-bit 高熵随机，bcrypt 反而过重且慢）。

### G. doc 03 错误码示例与 doc 20 完整字典关系不明【小】

doc 03 §3.3 列 11 条示例错误码，但 doc 20 §1.2 是完整枚举（80+ 条），关系未声明，AI Agent 可能误以 doc 03 为完整源。

### H. doc 21 Saga 步骤 state 名称与 doc 14 不一致【小】

doc 21 §4.1 Saga 中用 `pending`，但 doc 14 用 `new`；doc 21 用 `rejected` 不区分 validation/risk 失败，doc 14 区分得更细。

### I. M0-START 800 行口径未澄清【小】

doc 12 §1.3 规定 PR ≤ 800 行 diff（生成代码除外），但 M0 PR-3/PR-5 含较多装配代码（Makefile / docker-compose / CI yaml / Dockerfile），Agent 易误判超限。
