# 26 — 依赖与版本管理规范

> 状态：**强制规范** | 创建：2026-05-19 | 适用：全仓库

## 1. 核心原则（红线）

> **所有依赖、脚本、程序、语言版本号，全部使用官网最新的稳定版本（latest stable），除非存在以下明确的兼容性问题，否则不得妥协。**

允许保留旧版本的**唯一**理由：

1. **API/ABI 不兼容**：升级会破坏现有功能、需要大规模重构
2. **生态未跟进**：核心依赖（如 ConnectRPC、Buf、protoc-gen-*）不支持新版
3. **数据存储格式不兼容**：升级会导致数据迁移成本不可接受（如 PG 大版本）
4. **明确的运行时回归**：上游已确认的性能/正确性 regression

每条豁免必须在本文档 §4 中**列明原因 + 评估时间 + 重新评估周期**。

## 3. 升级流程

### 3.1 月度版本巡检（M-Bot）

每月第一个工作日，执行：

```bash
# Go
cd backend/go && go list -u -m all | grep '\['

# Node
cd frontend && pnpm outdated

# Python
cd research && uv pip list --outdated

# Docker images
docker compose -f deploy/docker-compose.prod.yml pull --policy=always
```

### 3.2 升级 PR 模板

每次依赖升级 PR 必须包含：

1. **变更类型**：patch / minor / major
2. **CHANGELOG 摘要**：上游 release notes 中的 breaking changes
3. **影响面评估**：调用此依赖的源文件列表
4. **回归测试结果**：`make test && make lint && make go-build`
5. **回滚方案**：旧版本号 + 回退命令

### 3.3 重大版本（major）升级

需经过：

1. 在 `docs/adr/` 添加 ADR
2. 创建升级分支，独立 PR
3. CI 全套通过 + 手动冒烟（`docker compose up`）
4. 至少 1 个 reviewer 确认

## 4. 当前已批准的版本豁免

> **必须列明每条豁免的过期日期。**

### 4.1 Go 1.26.0 — 当前已升级

- **现状**：`backend/go/go.mod` 声明 `go 1.26.0`，构建镜像 `golang:1.26-alpine`
- **完成日期**：2026-05-19
- **关闭豁免**

### 4.2 ConnectRPC Go v1（前端已 v2）

- **现状**：Go 后端 `connectrpc.com/connect v1.19.2`，前端 `@connectrpc/connect ^2.1.1`
- **原因**：Go 端 v2 尚未 GA（2026-05 时点）
- **风险**：协议层兼容（Connect 跨版本兼容）
- **重新评估触发**：v2 GA 发布后 30 天内
- **负责人**：后端组

### 4.3 Python 3.12（应升级到 3.13）

- **现状**：`research/pyproject.toml` 声明 `requires-python = ">=3.12"`
- **原因**：依赖生态（polars、scipy、optuna）部分版本对 3.13 兼容性需复测
- **重新评估触发**：完成依赖矩阵兼容测试后
- **负责人**：研究层

### 4.4 HashiCorp Vault 1.18（应升级到 2.0）

- **现状**：`deploy/docker-compose.prod.yml` 使用 `hashicorp/vault:1.18`
- **目标版本**：`hashicorp/vault:2.0`（2026-04-14 发布）
- **原因**：Vault 2.0 镜像入口脚本调用 `setcap` 设置 `CAP_IPC_LOCK`，需要 **CAP_SETFCAP** 内核能力；当前部署环境为 **WSL2**（`6.6.114.1-microsoft-standard-WSL2`），Docker Security Options 仅含 `cgroupns`，**不向容器赋予 CAP_SETFCAP**
- **已尝试无效方案**：
  - `cap_add: [IPC_LOCK]`
  - `VAULT_DISABLE_MLOCK=true` 环境变量
  - `command: server -dev -dev-disable-mlock`
  - `security_opt: [apparmor=unconfined, seccomp=unconfined]`
  - Docker daemon `seccomp-profile: ""` + `default-ulimits.memlock=-1`
- **结论**：WSL2 内核能力限制，非 compose 配置可解；属 Vault 2.0 的 breaking change（1.18 不调用 setcap）
- **风险**：无（dev mode 不涉及内存加锁的生产数据保护）
- **重新评估触发**：迁移到原生 Linux 主机后立即升级到 2.0
- **负责人**：DevOps / 运维

## 5. CI 守卫（自动化）

### 5.1 依赖审计

`.github/workflows/dep-audit.yml` 每周运行：

```yaml
- name: Go vuln check
  run: govulncheck ./backend/go/...

- name: Frontend audit
  run: cd frontend && pnpm audit --audit-level=high

- name: Python audit
  run: cd research && uv pip audit
```

### 5.2 镜像扫描

每次 build 后：

```bash
trivy image deploy-trading-core:latest --severity HIGH,CRITICAL
```

任何 HIGH/CRITICAL 漏洞必须在 7 天内修复或豁免（带 ADR）。

## 6. 升级时的破坏性数据迁移

### 6.1 PostgreSQL 大版本升级

**必须**：

1. `pg_dumpall` 全量备份
2. 启动新版本前，volume 必须使用 `pg_upgrade` 工具迁移
3. 不允许直接挂载旧 volume 到新主版本容器（会启动失败）

### 6.2 ClickHouse 大版本升级

ClickHouse 一般可以跨主版本兼容数据文件，但仍需：

1. 备份 `system.tables` 元数据
2. 升级前在 staging 环境验证 query 兼容性

## 8. 违反本规范的后果

- PR 不得合并
- 已合并的违规变更：发现后 24h 内回滚或补充豁免说明
- 长期未升级（超过 §4 截止日期）：阻塞下一次 release
